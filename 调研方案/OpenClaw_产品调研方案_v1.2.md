## Part 0：基准清单整理

> **性质**：桌面研究（desk research），不需要访谈或实测，1–3 天可完成。
>
> **目的**：在启动任何实测或访谈之前，先建立两份基准清单——调研对象是谁、已知能做什么。没有这两份清单，Part 1 的评测和 Part 2 的场景矩阵都无从入手。
>
> **数据来源**：官方文档、GitHub README 和 Issues、官网、Discord/Reddit/X 社区整理、YouTube/Bilibili 教程。

---

### 0.1 OpenClaw 生态项目清单

> **格式**：以 `openclaw_ecosystem_projects.csv` 为主记录文件，便于数据库追踪。每新增项目时，先使用 `openclaw_project_record_template.md` 完整填写，再将关键字段同步至 CSV。
>
> **与 AI 协作说明**：将项目记录模板连同已收集资料（官网截图、README、社区帖子）一并提供给 AI，要求按字段逐一填写并标注来源。AI 填写后需人工核查"已知限制"字段，确保有原文佐证。

**CSV 字段说明**：

| 字段 | 说明 |
|------|------|
| project_name | 项目名称 |
| type | 核心项目 / 托管类 / 基础设施类 / 安全类 / 垂直应用类 |
| region | 主要运营地区（全球 / 美国 / 中国 / 欧洲 / 中东 / 日韩，可多值） |
| description | 功能边界一句话摘要 |
| supported_scenarios | 已知支持场景（逗号分隔） |
| known_limitations | 主要限制（逗号分隔，需有来源） |
| messaging_platforms | 支持的消息平台：WhatsApp / Telegram / Discord / Signal / 微信 / **飞书** / **Slack** / LINE / 其他 |
| supported_llm | 支持的 LLM（逗号分隔） |
| active_status | 活跃 / 维护中 / 停止维护 |
| pricing_model | 免费 / 订阅 / 按量 / 一次性 / Freemium |
| notes | 其他备注 |

**按地区分节说明**：

- **全球**：无特定地区归属的项目（已预填 7 个已知项目）
- **美国**：在美国有实质运营或主要用户基础的项目
- **中国**：本土托管服务、本地 LLM 适配、微信/飞书/钉钉集成项目
- **欧洲**：有 GDPR 合规声明或欧洲服务器部署的项目
- **中东**：阿拉伯语支持、本地数据主权合规的项目
- **日韩**：LINE / Kakao 集成、本地语言 LLM 适配的项目

同一产品若在多个地区有实质运营，CSV 中 region 字段填多个值，模板文档中"地区特殊情况"节单独说明差异。

| region_us | 美国有实际使用记录：是 / 否 / 未知 |

| region_japan_korea | 日韩有实际使用记录：是 / 否 / 未知 |
| frequency_signal | 社区出现频率：高频 / 低频 / 单次 / 待调研 |
| required_skills | 所需核心 Skills（逗号分隔） |
| known_pain_points | 社区中提及的已知卡点（有原文佐证，逗号分隔） |

### 1.1 技术能力边界

#### 1.1.1 核心架构评估

通过阅读源码、部署测试、Issue 分析，回答以下问题：

| 架构层 | 评估维度 | 需要回答的问题 |
|--------|----------|---------------|
| Gateway daemon | 稳定性与并发 | 高并发下（50+ 会话）是否出现连接丢失？重启后状态恢复机制？ |
| Skills 系统 | 能力扩展性 | SKILL.md 声明式设计的上限在哪里？复杂跨系统任务是否有结构性缺陷？ |
| 消息平台集成 | 交互表达力 | 消息平台 UI 对复杂任务的表达和结果展示有哪些天然限制？ |
| 记忆系统 | 上下文持久化 | 长对话/跨会话的记忆准确率如何？记忆污染问题是否存在？ |
| 本地模型运行 | 性能与门槛 | llama.cpp/MLX/ONNX 各引擎的实测延迟；最低可用硬件配置 |

**调研方法**：源码审计 + 本地部署实测 + GitHub Issue 分类统计（按组件标记问题频率）

---

#### 1.1.2 Agent 可靠性基准测试

设计标准任务集，实测任务完成率（TCR）和失败模式，作为 Part 4 技术约束的依据。

**任务分层设计**：

| 层级 | 任务特征 | 示例 | 测试目标 |
|------|----------|------|----------|
| L1：单步执行 | 单工具调用，无条件分支 | 发送消息、读取文件、查询 API | 建立基础可靠性基线 |
| L2：线性多步 | 3–5 步顺序执行，无错误处理 | 读取邮件→摘要→起草回复 | 评估步骤间状态传递准确性 |
| L3：条件分支 | 含 if/else 逻辑，需模型判断 | 判断邮件优先级→分类处理 | 评估模型推理在执行链中的稳定性 |
| L4：跨系统协作 | 多工具跨平台，含错误恢复 | 日历冲突检测→重新安排→通知相关方 | 评估错误处理和恢复机制 |
| L5：长时任务 | 执行时间 >5 分钟，含状态持久化 | 竞品信息收集→整理→生成报告 | 评估任务中断和续跑能力 |

**每类任务各执行 20 次，记录**：
- 任务完成率（TCR）
- 首次成功率（无需用户干预）
- 失败类型分布（模型推理错误 / 工具调用失败 / 权限问题 / 超时）
- 平均 token 消耗
- 端到端延迟（P50 / P95）

**卡点根因分类**（每个失败案例需归因）：

| 根因类型 | 描述 | 产品含义 |
|----------|------|----------|
| 模型推理不稳定 | 相同输入不同输出，幻觉，步骤遗漏 | 需要输出验证层或人工确认节点 |
| Skill 设计缺陷 | SKILL.md 表达能力不足，参数传递错误 | 需要更结构化的 Skill 规范 |
| 权限与沙箱限制 | 所需权限被拒，沙箱隔离过强 | 需要细粒度权限配置 UI |
| 消息 UI 表达瓶颈 | 用户无法在消息界面清晰表达复杂意图 | 需要引导式输入或任务模板 |
| 错误处理缺失 | 无重试机制，失败后无明确反馈 | 需要错误处理和恢复流程设计 |
| 本地模型能力差距 | 本地模型在复杂任务上与云端差距大 | 需要明确本地 vs 云端的任务分配策略 |

---

#### 1.1.3 安全边界评估

| 评估维度 | 调研内容 | 产品决策依据 |
|----------|----------|-------------|
| Prompt Injection 风险 | 实测：恶意内容能否通过 Skill 或消息内容劫持 Agent 行为 | 哪些场景不能做无人值守自动化 |
| Skills 沙箱隔离 | 当前是否有进程/容器级隔离？Skills 的系统访问范围 | 企业场景的技术可行性前提 |
| 敏感数据处理 | Agent 执行链中敏感数据是否会意外出现在 LLM 上下文中 | 医疗/法律/金融场景的合规约束 |
| 本地 vs 云端数据流 | 哪些数据留本地，哪些发往 LLM API | 隐私敏感用户的产品设计约束 |

### 1.2 各地区应用现状

> **每个地区统一使用以下结构**，数据来源：社区访谈 + Reddit/Discord/X 内容分析 + 本地论坛（少数派、V2EX 等）+ 产品使用数据

**地区分析模板**：

```
1. 主要用户群体：谁在用，背景特征
2. 主流使用场景：实际在用 OpenClaw 做什么
3. 体验质量评估：好用在哪里，不好用在哪里
4. 卡点清单：按频率排序的具体卡点（需有原话或截图佐证）
5. 卡点根因：对照 1.1.2 的根因分类，判断每个卡点的技术/产品/本地化根因
6. 本地化特殊因素：监管、支付习惯、语言、文化、竞品生态的特殊性
```

---

#### 美国

**数据来源重点**：OpenClaw Discord、Reddit（r/selfhosted、r/LocalLLaMA）、X（@openclaw 提及）、Product Hunt 评论

**重点调研问题**：
- 技术用户和非技术用户的比例分布
- 哪些 Skills 安装量最高？为什么
- 用户在哪一步放弃首次设置
- 已有创业公司（KiloClaw、Agent 37 等）的用户是否流失，流失原因
- 企业用户（小微企业主、自由职业者）的典型工作流

---

#### 中国

**数据来源重点**：少数派、V2EX、即刻、微信/飞书社群、GitHub Issues（中文）、Bilibili 教程评论

**重点调研问题**：
- 国内用户如何解决 LLM API 访问问题（中转、本地模型、境外账号）
- 国内消息平台（微信、飞书、钉钉）与 OpenClaw 集成的实际情况——这些平台的 Bot API 限制更严
- 企业数据出境合规是否是实际使用障碍
- 国内竞品（Coze、扣子、字节/阿里系 Agent 产品）的功能重叠度
- 非技术用户的上手失败率和主要原因

**本地化特殊因素**：
- 微信 API 对 Bot 的严格限制，影响消息平台 UI 的核心价值主张
- 生成式 AI 备案要求对商业化产品的约束
- 本地模型（Qwen、DeepSeek）的实际表现 vs 国际模型

---

#### 欧洲

**数据来源重点**：GitHub Issues（德/法语）、Hacker News 欧洲用户、本地技术社区

**重点调研问题**：
- GDPR 下数据本地化需求是否是真实驱动力
- 企业用户对"数据不离开设备"的实际支付意愿
- OpenClaw 的本地模型支持是否满足隐私合规需求
- 欧洲用户对消息平台的使用习惯（WhatsApp 渗透率高，但 Signal 有一定比例）

**本地化特殊因素**：
- GDPR 合规是产品壁垒，也是护城河
- 企业级 AI 工具的采购决策链更长、更谨慎
- 语言多样性（德/法/西/意）对 LLM 表现有影响

---

#### 中东

**数据来源重点**：X/Twitter（阿语/英语）、LinkedIn（海湾国家科技从业者）、Reddit（r/saudiarabia tech）

**重点调研问题**：
- 当地 AI Agent 工具的渗透程度（基准线较低）
- 高净值个人用户和企业用户的具体使用场景
- 阿拉伯语支持对 LLM 任务质量的影响
- 本地消息平台习惯（WhatsApp 主导）
- 政府和企业数据主权要求

**本地化特殊因素**：
- 人均支付能力高，但技术成熟度差异大
- 沙特、UAE 等国有主动推进 AI 国家战略，政府采购可能是渠道
- 阿拉伯语 RTL 界面和本地化 LLM 质量是重要约束

---

#### 日韩

**数据来源重点**：Qiita/Zenn（日）、GitHub Issues（日语）、韩国技术 Discord、Naver 论坛

**重点调研问题**：
- 日本用户对自动化 Agent 的信任度和接受程度（文化因素）
- 韩国创业生态中是否有 OpenClaw 类工具的使用案例
- 本地 LLM（日：Rinna、Elyza；韩：CLOVA X）的整合可能性
- 企业 IT 决策链特点（日本企业层级决策慢，韩国科技公司相对灵活）

**本地化特殊因素**：
- 日本用户对出错的容忍度极低，需要更高的 TCR 标准
- 日语/韩语 LLM 表现与英语有差距，影响推理类任务质量
- 韩国 Kakao、日本 LINE 是主流消息平台，集成难度需评估

## Part 2：Agent 应用场景矩阵

### 2.1 Agent 理论任务全景图

按任务类型对 Agent 理论上能做的事进行完整分类，不预设产品边界。

| 一级类型 | 二级类型 | 任务特征 | 典型例子 |
|----------|----------|----------|----------|

| | 被动监听 | 持续监控并在条件触发时响应 | 邮件筛选、价格预警 |
| | 文档处理 | 解析、提取、转化非结构化数据 | 合同信息提取、简历解析 |
| **跨系统自动化** | 数据同步 | 在多个系统间传递和转换数据 | CRM 更新、日历同步 |
| | 流程编排 | 按既定流程跨工具执行多步任务 | 报销审批、入职流程 |
| | 内容发布 | 生成并发布到目标平台 | 社媒发帖、邮件发送 |
| **决策辅助** | 分析汇总 | 收集数据并提炼洞察 | 周报生成、销售漏斗分析 |
| | 方案生成 | 基于约束条件给出可选方案 | 日程规划、路线推荐 |
| | 风险提示 | 识别异常并告警 | 合规风险扫描、预算超支预警 |
| **交互代理** | 对话代理 | 代用户与他人/系统交互 | 客服响应、会议记录 |
| | 任务委托 | 接收并转派子任务 | 项目管理助手 |
| **个人知识管理** | 记忆外挂 | 结构化存储用户上下文 | 笔记整理、偏好记忆 |
| | 主动提醒 | 基于上下文主动发起动作 | 跟进提醒、习惯追踪 |

### 2.2 各场景实际渗透率与竞品覆盖

**评估框架**（每个场景填写）：

| 字段 | 内容 |
|------|------|
| 场景名称 | — |
| 渗透率评级 | 高（>30% 目标用户在用 AI）/ 中（5–30%）/ 低（<5%）|
| 主要竞品 | 列出 2–4 个代表性工具 |
| 竞品完成质量 | 优秀 / 可用 / 粗糙 / 无法完成 |
| 用户满意度信号 | 社区评价、NPS 信号、投诉频率 |
| 核心未满足需求 | 现有方案最大的缺口是什么 |

**重点调研的 Agent 竞品**（覆盖主要赛道）：

| 产品 | 定位 | 重点评估场景 |
|------|------|-------------|
| Claude Computer Use | 桌面控制 Agent | 跨系统自动化、流程编排 |
| AutoGPT | 开源通用 Agent | 信息获取、长时任务 |
| Zapier AI | 工作流自动化 + AI | 数据同步、流程编排 |

### 2.4 优先级矩阵推导

将 2.1 / 2.2 / 2.3 的数据叠加，生成优先级矩阵。

**推导逻辑**：

```
场景价值评分 = 渗透潜力 × 需求强度 × OpenClaw 结构性优势 × (1 / 竞争饱和度)
```

**象限定义**：

| 象限 | 渗透率 | 竞品质量 | OpenClaw 覆盖 | 结论 |
|------|--------|----------|--------------|------|
| **优先做** | 低–中 | 粗糙或无 | 可行 | 市场空白 + 技术可行，重点投入 |
| **差异化做** | 中–高 | 已有但有缺口 | 可行 | 需要明确差异化角度才能进入 |
| **观察** | 低 | 粗糙 | 部分可行 | 需等技术成熟或市场教育 |
| **不做** | 高 | 优秀 | — | 竞争饱和，OpenClaw 无结构性优势 |
| **排除** | 任意 | 任意 | 结构性不可行 | 技术约束导致 |

**输出**：按优先级排序的场景列表，每个场景附优先级依据（1–2 句话）。这个列表是 Part 3 用户研究的输入——只对"优先做"和"差异化做"的场景设计访谈。

### 3.2 访谈指南

> 以下是通用访谈结构，每个场景需在「场景专项探索」部分补充具体问题。

#### 模块一：背景与上下文（5 分钟）

- 你平时用 AI 工具主要做什么？大概从什么时候开始用？
- 你是怎么知道 OpenClaw 的？第一次用是什么时候？

#### 模块二：使用行为还原（15 分钟）

> 目标：还原真实的使用过程，而非听用户总结评价。

- 上次用 OpenClaw 完成了什么任务？你能描述一下整个过程吗？（逐步追问）
- 在这个过程中，哪一步你等待的时间最长？
- 哪一步你最不确定 AI 会不会做对？
- 有没有哪个地方你本来想让它做，但最后没让它做——为什么？
- 这个任务最终有没有完成？如果没有，是在哪一步停下来的？

#### 模块三：场景专项探索（15 分钟）

> 根据 Part 2 的具体场景补充，例如：

**如果是「信息获取与监控」场景**：
- 你目前是怎么追踪（某类信息）的？AI 介入前你是怎么做的？
- 你希望 AI 收集完信息后直接告诉你结论，还是给你原始数据？为什么？
- 如果它拿回来的信息有 10% 是错的，你能接受吗？如何发现错误？

**如果是「跨系统自动化」场景**：
- 你现在跨多少个工具完成这个任务？哪个工具之间的衔接最让你头疼？
- AI 执行这个任务时，你觉得哪些步骤你必须自己确认，不放心让它自动做？
- 任务执行到一半失败了，你怎么知道失败了？失败后你怎么处理？

#### 模块四：卡点深挖（10 分钟）

> 针对模块二中出现的卡点逐一追问根因。

- 你说这里会出错——上次出错的时候，具体发生了什么？
- 你觉得这个问题是 AI 不够聪明，还是这个工具设计有问题，还是你没说清楚？
- 如果这个问题解决了，你会更频繁地用这个功能吗？频繁多少？

#### 模块五：付费意愿与价值感知（5 分钟）

- 你现在用 OpenClaw 是免费还是付费？
- 如果有一个产品能把刚才说的问题都解决，你愿意付多少钱？（追问：为什么是这个数字？）
- 你现在不用 AI 的话，这件事是怎么解决的？花多少时间/钱？

### 4.3 交互流程设计框架

> Agent 产品的交互设计需要同时描述**用户侧**（用户看到什么、做什么）和**Agent 侧**（Agent 在做什么、状态如何）。

#### 双轨交互流程模板

```
用户侧                          Agent 侧
──────────────────────────────────────────────────────
[用户输入意图]                →  [意图解析 + 任务规划]
                              ←  [展示执行计划，等待确认]
[用户确认 / 修改]             →  [开始执行]
                              ←  [进度更新：正在做第 X 步]
                              ←  [遇到需要确认的节点：展示选项]
[用户做出选择]                →  [继续执行]
                              ←  [任务完成：展示结果摘要]
[用户审核结果]                →  [用户接受 / 要求修改]
```

#### 关键设计决策点

每个交互流程需要对以下问题做出明确决策，不能模糊处理：

| 决策点 | 需要回答的问题 |
|--------|---------------|
| **透明度** | 用户需要看到 Agent 的执行步骤吗？看多少细节？ |
| **确认粒度** | 哪些步骤需要确认？确认太多影响体验，太少影响信任 |
| **进度呈现** | 长时任务（>30 秒）如何让用户感知进度？ |
| **失败呈现** | 任务失败时展示什么？用户能做什么？ |
| **撤销机制** | 哪些操作可以撤销？撤销的时间窗口是多少？ |
| **通知策略** | 异步任务完成时如何通知用户（哪个渠道、什么时机）？ |

#### 地区差异说明

交互流程需要在以下维度标注地区差异：
- **确认节点**：日本用户对出错容忍度低，需要更多确认节点；美国用户更接受自动化
- **语言与 LLM**：中文/阿拉伯语/日语任务需要评估对应语言 LLM 的能力差距，影响 TCR
- **消息平台**：中国的微信限制、日本的 LINE 接口特性，影响消息 UI 的表达能力

### 4.5 设计原则提炼方法

> 设计原则不是凭空写出来的，而是从卡点根因中推导出来的。以下是推导流程。

#### 推导流程

```
卡点（来自 3.3 步骤一）
    ↓ 追问：这个卡点反映了什么深层设计问题？
根因（对照 1.1.2 分类）
    ↓ 追问：如果要从设计层面缓解这个根因，核心原则是什么？
设计原则（1 句话，行为导向）
    ↓ 落地到：哪些功能/流程应用这条原则？
具体设计决策
```

#### 示例推导

```
卡点：「用户不知道任务执行失败了，等了很久才发现」（出现 6/10 次访谈）
根因：错误处理缺失 + 进度状态不透明
设计原则：「Agent 的任何状态变化，用户都应在 30 秒内得到反馈」
具体决策：
  - 执行超过 10 秒的任务必须展示进度条或步骤进展
  - 任务失败必须通过消息主动告知用户，不依赖用户主动查询
  - 失败消息必须包含：失败原因（非技术语言）+ 用户可做的下一步
```

#### 设计原则记录格式

```markdown
## 原则名称：[简洁命名，如「状态透明」]

### 来源
- 卡点：[来自哪个访谈，什么场景]
- 出现频率：[X/Y 次访谈]
- 根因类型：[对照 1.1.2 分类]

### 原则陈述
[一句话，行为导向，可测试]

### 适用范围
[哪些功能/流程需要遵守这条原则]

### 反例（什么是违反这条原则的）
[具体描述，帮助团队理解边界]

### 优先级
高 / 中（高 = 违反会直接损害信任或导致放弃使用）
```
