# 模块 1：产品定位与竞争策略

## 一句话总结

OpenClaw 是一个跑在用户自己设备上的个人 AI 助手控制中枢，通过统一接入 23+ 消息渠道、共享记忆和可编程工具系统，构建了一个"有持续记忆的 AI 人格底座"。

---

## 产品定位的三层理解

| 层次 | 定位 | 用户感知 | 竞品对比 |
|------|------|---------|---------|
| **表层** | 多渠道 AI 聊天工具 | "我能在 WhatsApp 上用 Claude！" | 类似 Poe，但自托管 |
| **中层** | 本地化 AI 助手，能执行真实任务 | "它能帮我操作电脑、定时跑任务" | 类似 ChatGPT Desktop，但更强大且开放 |
| **深层** | 个人 AI 基础设施——有持续记忆、可编程、可扩展的 AI 人格底座 | "它比任何人都了解我的工作习惯" | **没有直接竞品** |

表层吸引人进来，中层留住人，深层锁住人。

---

## 与竞品的本质区别

| | ChatGPT | OpenClaw |
|---|---------|---------|
| 在哪里跑 | OpenAI 的服务器 | 你自己的电脑 |
| 怎么跟它聊 | 打开网页/App | 在你已经用的任何聊天软件里发消息 |
| 能做什么 | 聊天、简单代码执行、搜索 | 控制浏览器、读写文件、执行命令、定时任务 |
| 数据在哪 | OpenAI 的云端 | 你自己的磁盘 |
| 能换模型吗 | 只能用 OpenAI 的模型 | 30+ 模型提供商 |
| 扩展性 | GPT Store（受限） | 82 个开源插件 + 第三方生态 |

**核心差异不是"功能更多"，而是定位完全不同：ChatGPT 是产品（你是用户），OpenClaw 是基础设施（你是主人）。**

---

## 关键设计决策与产品哲学

### 1. "自己跑"是有意的用户筛选，不是技术限制

创始人在 VISION.md 中明确表示：终端优先是故意的设计，让用户看到认证、权限和安全配置。不做"一键安装"的便利包装，因为 AI 助手能执行代码、控制浏览器——如果用户不理解权限，比不装还危险。

**产品哲学：先安全，后便利。** 先筛选出理解风险的用户群，安全模型成熟后再降低门槛。

**启发：AI 产品的"安装门槛"和"安全门槛"是一体的。** 降低安装门槛很容易，但安全模型没跟上就是给用户挖坑。

### 2. 23+ 渠道是增长飞轮，不是功能堆砌

每加一个渠道 = 打开一个地域/人群的入口：
- WhatsApp → 东南亚、拉美、欧洲
- Telegram → 技术圈、加密圈
- Slack/Teams → 企业用户
- 飞书 → 中国企业
- LINE → 日本、泰国
- Zalo → 越南

渠道由社区贡献（Zalo 插件是越南开发者写的，飞书插件是中国开发者写的），OpenClaw 只需要插件 SDK 足够好。

**这是典型的平台型增长策略：用渠道覆盖度做分发。**

### 3. 共享记忆是真正的护城河

**关键架构决策：Session 和 Memory 跟 Agent 绑定，不跟渠道绑定。**

```
WhatsApp 上的你 ──┐
Telegram 上的你 ──┤──→ 同一个 Agent ──→ 同一份记忆
Slack 上的你 ────┘
```

这意味着：
- 你在 WhatsApp 上告诉 AI"我喜欢简洁的回复"，切到 Slack 上它也记得
- 你在 Telegram 上分析了一份报告，在 iMessage 上问"刚才那份报告的结论"，它能答

**这不是"多渠道聊天工具"，而是"一个有持续记忆的 AI 人格，碰巧可以从任何渠道触达"。**

产品公式：**渠道是触点，Agent 是实体，记忆是护城河。**

## 竞争格局分析

### 巨头的结构性盲区

大公司做的托管版天然有"锁定"属性：

| 大公司托管版 | 锁定方式 |
|------------|---------|
| Apple Intelligence | 锁死 Apple 生态，只用 Apple 选的模型 |
| Google Gemini | 数据喂给 Google，广告模型的一部分 |
| 微软 Copilot | 绑定 Microsoft 365，企业定价 |
| ChatGPT | 只用 OpenAI 模型，数据在 OpenAI 服务器 |

**巨头的战略目标 = 把用户锁在自己的生态里 → 不会真心做跨平台 → 跨平台需求永远存在但巨头不愿满足 → 小公司的结构性机会。**

历史验证：Slack（跨平台团队协作）、1Password（跨平台密码管理）、Notion（跨平台知识管理）——都是利用巨头的"跨平台真空"成长起来的。

### 界面与模型正在分离——市场比想象的大

```
2023 年：用户 → ChatGPT 网页 → GPT 模型（界面和模型绑死）
2025 年：用户 → Cursor/Claude Code/OpenClaw/企业工具 → 任意模型 API（分离了）
```

越来越多的人不是在"用 ChatGPT 网页"，而是通过各种工具调用 API。这意味着：不需要"黑入"闭源产品注入记忆，而是自己做界面/中间层，调 API，记忆层天然就是你的。

OpenClaw 的做法：调用 Claude/GPT API → 调用前把记忆注入 prompt → 调用后从对话中提取新记忆。模型厂商不知道也不在乎你在 prompt 里塞了什么。

## 创业机会分析

### 跨平台 AI 产品的商业模式风险

跨平台的价值和脆弱性来自同一个地方——依赖多个平台：
- **平台封杀风险**：WhatsApp 用非官方库（Baileys），微信对非官方接入零容忍
- **平台自己下场**：Apple Intelligence、微软 Copilot 都在做
- **你在帮平台增值，平台不会感谢你**

**结论：商业模式不能建立在"连接渠道"这一层，因为太脆弱。要建在渠道之上、用户难以迁移的层面。**

### 三种可行的商业模式

#### 模式 1："AI 人格" SaaS（最适合小公司起步）

核心卖点不是"跨平台"，而是"一个始终了解你的 AI"。跨平台只是实现手段。

**用户体验：**
- 第一天：AI 开始建立用户画像
- 第一周：AI 积累偏好（工作内容、回复风格、活跃时间）
- 第二周：用户接入第二个渠道，发现 AI 已经了解他——跨渠道记忆打通
- 第一个月：用户离不开了——不是因为功能，而是因为 AI 积累了关于他的记忆

**定价设计：**
- 免费版：1 渠道 + 每天 50 条 + **7 天记忆**（超过自动遗忘）
- 专业版 ¥49/月：3 渠道 + 无限对话 + 永久记忆 + 自定义人设
- 团队版 ¥199/月：无限渠道 + 共享知识库 + 多角色 + 权限管理

**关键：免费版的"7 天记忆"是最强付费转化。** 用户体验过"AI 记得我"后，回不去了。

**产品架构：**
```
触达层（渠道接入）← OpenClaw 的 Channel 抽象
    ↓
AI 人格层（核心差异化）← OpenClaw 的 Memory + Context Engine
  用户画像 / 对话风格 / 知识积累 / 习惯模式
    ↓
模型调用层（可替换）← OpenClaw 的 Provider 插件系统
```

**护城河：**
- 第 1 层：记忆数据（用 3 个月后换产品 = 从零教 AI）
- 第 2 层：使用习惯（改变工作流的摩擦力）
- 第 3 层：团队网络效应（共享知识库，一人离开 = 失去共建知识）

**切入点小**：先做微信/飞书覆盖中国市场，不需要一上来 23 个渠道。

**MVP 四件事：** 一个渠道 + 一个模型 API + 一个记忆系统 + 一个用户画像自动构建。

#### 模式 2："Agent 能力"按量计费

```
免费：AI 聊天
按量付费：工具调用 $0.01/次 · 定时任务 $5/月 · 多 Agent $20/月 · 高级记忆 $10/月
```

类似 Zapier 模式。用得越多付得越多，用得越多越难离开。

#### 模式 3：开源 + 企业私有化部署

```
小团队：托管版 SaaS $50/月
中型企业：私有部署 + 数据隔离 $500/月
大企业：On-premise + 定制集成 + SLA 年费
```

经典开源商业模式（类似 GitLab）。

### 记忆层作为独立基础设施的机会

不局限于自部署场景。潜在客户有三类：

| 客户类型 | 痛点 | 怎么卖 |
|---------|------|--------|
| **个人用户** | ChatGPT 记不住我是谁 | toC：带记忆的 AI 助手 |
| **AI 应用开发者** | 做 AI 客服/教练/写作工具，需要记忆能力不想自己造 | toD：记忆层 SDK/API |
| **企业** | 内部 AI 工具没有记忆，员工每次重复解释背景 | toB：企业记忆层 SaaS |

**AI 应用开发者可能是最甜的市场**——无数团队在做垂直 AI 产品（客服、教练、写作、编程、心理咨询），每个人都需要记忆能力，没人想从零搭建。

**MCP（Model Context Protocol）正在标准化这个接口**——未来任何支持 MCP 的 AI 应用都可以即插即用地接入记忆层服务。

```
市场定位：
  模型层（大厂的事）
      ↑ 调用
  应用层（垂直团队做）
      ↑ 接入
  ★ 记忆层（你的位置）★
    通过 SDK / API / MCP 提供服务
```

### 主要风险

1. **渠道封杀**：微信零容忍非官方接入，先从 Telegram/飞书等开放平台切入
2. **模型能力趋同**：但记忆不是模型的能力，是产品的能力——模型是工具，记忆是数据资产
3. **付费意愿待验证**：先做免费版跑三个月，看留存和反馈再定价

## 关键术语表

| 术语 | 含义 |
|------|------|
| Gateway | OpenClaw 的核心进程，跑在本地，所有渠道和客户端通过 WebSocket 连接它 |
| Agent | 一个 AI 助手实例，有独立的记忆、人设、工具权限和会话 |
| Channel | 消息渠道（WhatsApp、Telegram、Slack 等），通过插件接入 |
| Session | 一轮或多轮对话的上下文，跟 Agent 绑定而非渠道绑定 |
| Memory | 长期记忆，从对话中提取并持久化存储，支持语义检索 |
| Workspace | Agent 的"大脑"目录（AGENTS.md/SOUL.md/USER.md），git 友好 |
| Plugin/Extension | 可插拔的扩展模块，覆盖渠道、模型提供商、工具、记忆后端 |
| Provider | 模型提供商（Anthropic、OpenAI、Google 等），通过 Extension 接入 |
| MCP | Model Context Protocol，Anthropic 推的开放协议，标准化 AI 应用的上下文和工具接入 |
| Binding | 路由绑定——把特定渠道/用户/群组映射到特定 Agent |

# 模块 2：架构全景

## 一句话总结

OpenClaw 是一个六层架构的系统：触达层接用户、网关层管连接、路由层分消息、智能层跑 AI、能力层做真事、基础设施层保稳定。模型调用只占整个工作量的约 10%，剩下 90% 是让"跟 AI 聊天"达到生产级所需要的一切。

---

## 架构地图：六层全景

```
第 1 层 触达层      用户从哪来          23+ 渠道 + Web + App + CLI
第 2 层 网关层      流量怎么管          单进程 + 插件加载 + 健康监控
第 3 层 路由层      消息交给谁          绑定规则选 Agent + 会话范围选 Session
第 4 层 智能层      AI 怎么想          上下文组装 + 执行循环 + 记忆 + 多 Agent
第 5 层 能力层      AI 能做什么         工具 + 媒体 + 沙箱（8 层权限过滤）
第 6 层 基础设施层   怎么稳定跑         配置 + 插件 SDK + 安全 + 守护进程
```

---

## 第 1 层：触达层

### 四种触达方式

| 方式 | 谁在用 | 怎么连 |
|------|--------|--------|
| 23+ 消息渠道 | 普通用户 | 插件连接器（Telegram 长轮询/WhatsApp WebSocket/Slack Bolt…） |
| WebChat 网页 | 不想装东西的用户 | 浏览器直连 |
| 客户端 App（macOS/iOS/Android） | 想要语音唤醒、摄像头等深度功能的用户 | WebSocket |
| CLI 命令行 | 开发者 | 本地直连 |

### 关键设计决策

**"不做自己的界面，寄生在用户已有的界面里"**——这是一个分发策略选择：
- 不需要说服用户下载新 App（用户已经有 WhatsApp/Telegram/Slack）
- 获客成本极低（"把 Bot 加到你的 Telegram"比"下载新 App"容易 10 倍）
- 四种触达方式不是四个产品，而是同一个 Gateway 的四个入口，共享 Agent、记忆和工具

---

## 第 2 层：网关层

### 核心职责

| 职责 | 做什么 |
|------|--------|
| 生命周期管理 | 启动/重连/健康检查每个渠道连接器 |
| 认证与安全 | Gateway 本身的 Token 认证 + 各渠道凭证管理 |
| 流量控制 | 速率限制 + 泳道（Lane）并发隔离 + 队列管理 |
| 热更新 | 配置变更不用重启 Gateway |

### 关键设计决策

**"单进程 + 插件加载"而非"每渠道一个微服务"**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 微服务 | 隔离好 | 占内存多、运维复杂、个人用户搞不定 |
| 单进程 + 插件 | 一条命令启动、资源少 | 一个插件崩可能影响其他 |

OpenClaw 选了后者，用工程手段弥补——每个连接器有独立错误边界、健康检查和自动重启（指数退避 + 抖动防雪崩）。

**产品启发：架构选择要看用户是谁。** 个人用户 → 单进程是唯一正确答案。不是"什么架构更先进"，而是"什么架构适合你的用户"。

## 第 3 层：路由层

### 两步走：先选 Agent，再选 Session

```
消息到达
  → 第一步：路由（交给哪个 Agent？）→ 查绑定规则
  → 第二步：会话范围（放进哪个 Session？）→ 看 Agent 配置
```

### 第一步：路由绑定——查找顺序（从具体到宽泛）

用户在配置文件中定义绑定规则（Bindings），系统按优先级匹配：

```
① peer（具体的人）         "张三的消息 → 交给 work Agent"
② parent peer（上级身份）
③ guild + role（群 + 角色） "#开发群的管理员 → 交给 code Agent"
④ guild（群）              "#开发群 → 交给 code Agent"
⑤ account（渠道账号）       "我的 Telegram Bot → 交给 life Agent"
⑥ 默认 Agent               "交给 main"
```

类比：公司前台转接电话。先看谁打的 → 再看打到哪条线 → 最后默认转总机。越具体的规则优先级越高。

**只在有多个 Agent 时才有意义。** 只有一个 Agent 则所有消息都交给它。

### 第二步：会话范围——三种模式

确定 Agent 后，决定消息放进哪个 Session（对话历史）：

| 模式 | 效果 | 适合场景 |
|------|------|---------|
| `main` | 所有消息合并到一个 Session | 个人独用，跨渠道共享上下文 |
| `per-sender` | 每个发送者独立 Session | 一个 Bot 服务多人，各聊各的 |
| `per-channel-peer` | 渠道 × 发送者 二维隔离 | 严格场景隔离（工作/生活分开） |

### 关键概念：Agent ≠ Session

```
Agent = 一个 AI 人格（人设、工具权限、模型、记忆库）→ 类比：一位医生
Session = 一轮对话上下文（对话历史、当前话题）→ 类比：一份病历

一个 Agent 可以有很多 Session。
会话范围控制的不是"新起 Agent"，而是"新起 Session"。
```

### 产品启发

会话范围本质上是**便利性 ←→ 隐私性的滑块**。OpenClaw 没有替用户做选择，而是把选择权给了用户。`main` 模式实现了"共享底层记忆"的理想体验，`per-sender` 保障了多人场景的隐私隔离。

---

## 第 4 层：智能层（核心）

智能层内部有四个子系统，后续模块 4-7 会逐一深入拆解：

| 子系统 | 职责 | 深入模块 |
|--------|------|---------|
| **上下文工程** | 组装喂给模型的信息（System Prompt + 记忆 + 历史 + 工具列表） | 模块 7 |
| **Agent 执行引擎** | 调用模型 → 执行工具 → 循环直到最终回复 + 失败重试/模型切换 | 模块 4 |
| **记忆系统** | 短期会话记忆 + 长期语义记忆，跨对话延续 | 模块 6 |
| **多 Agent 编排** | 复杂任务分工，Subagent 本地派生 + ACP 隔离派生 | 模块 5 |

**它们的协作关系：**
- 上下文工程负责**喂什么**给模型
- Agent 引擎负责**跑循环**
- 记忆系统负责**跨对话的延续性**
- 多 Agent 负责**复杂任务的分工**

**产品启发：模型调用只是中间一步。真正决定 AI 回复质量的是模型看到了什么（上下文工程）和做完之后记住了什么（记忆系统）。这两层才是产品差异化的地方。**

---

## 第 5 层：能力层

### 工具系统

| 类别 | 工具 | 能力 |
|------|------|------|
| 文件 | read / write / edit | 读写用户文件 |
| 执行 | exec / process | 跑 shell 命令 |
| 浏览器 | browser snapshot / navigate | 控制 Chrome |
| 搜索 | web_search / web_fetch | 搜索互联网 |
| 消息 | message_send | 发消息给人 |
| 媒体 | image_generate / tts | 画图、说话 |
| 调度 | cron | 定时任务 |

### 工具权限：8 层过滤管道

每个工具在每个场景下能不能用，经过 8 层过滤：

```
① 工具配置文件（minimal/coding/messaging/full）
② 模型提供商的工具策略
③ 全局允许/禁止列表
④ Provider 级策略
⑤ Agent 级策略
⑥ Agent Provider 策略
⑦ 沙箱策略
⑧ 子 Agent 策略
```

**设计哲学：能力越大，控制越细。** 不是简单的开/关，而是根据谁在用、什么环境、哪个 Agent 来层层过滤。

### 媒体管道

- **输入**：图片 → 视觉识别 / 语音 → 转文字 / 视频 → 抽帧分析
- **输出**：图片生成 / 语音合成 / Canvas 交互界面
- 支持多提供商自动降级（OpenAI Vision → Google Vision → Deepgram）

### 沙箱隔离

三种执行环境，自动降级：Docker（最安全） → systemd-run（Linux 轻量） → 原生（信任模式）

# 模块 3：一条消息的一生

## 一句话总结

一条消息从用户按下发送键到收到 AI 回复，经历六个阶段：到达 → 缓冲去重 → 归一化与权限 → 路由 → 上下文组装与 Agent 执行 → 回复出站。模型调用只是第五阶段里的一个环节，前后大量工作决定了 AI 是否能给出高质量回复。

---

## 场景

> 用户在 Telegram 私聊里发送："帮我总结今天的科技新闻"
> 几秒后，AI 回复了一段新闻摘要。

---

## 全流程图

```
用户在 Telegram 发送 "帮我总结今天的科技新闻"
 │
 ▼
阶段 1 ─ 消息到达
 │  Telegram 连接器轮询拿到消息
 ▼
阶段 2 ─ 缓冲去重
 │  去重 → 碎片合并 → 防抖
 ▼
阶段 3 ─ 归一化 + 权限
 │  Telegram 格式 → MsgContext → 检查 DM 权限
 ▼
阶段 4 ─ 路由
 │  绑定规则 → main Agent → Session Key
 ▼
阶段 5 ─ 上下文组装 + Agent 执行
 │  System Prompt + 记忆 + 历史 → 调模型
 │  → 模型决定搜索 → 执行 web_search
 │  → 拿到结果 → 再调模型 → 生成摘要
 │  → 流式输出 → 记忆更新
 ▼
阶段 6 ─ 回复出站
 │  检查字数限制 → 流式推送草稿 → 发送最终消息
 ▼
用户在 Telegram 看到新闻摘要
```

---

## 阶段 1：消息到达

### 发生了什么

Telegram 连接器通过长轮询（getUpdates API），每隔几百毫秒向 Telegram 服务器询问"有没有新消息"。收到新消息后进入处理流程。

### 关键设计决策：轮询 vs Webhook

| 方案 | 原理 | 要求 |
|------|------|------|
| Webhook | Telegram 主动推消息到你的服务器 | 需要公网可访问的 URL |
| 轮询 | 你主动去 Telegram 拉消息 | 只需要能访问互联网 |

OpenClaw 选轮询，因为目标用户在自己笔记本上跑，大多数人没有公网 IP。不需要配域名、SSL、开端口。

**产品约束"用户是个人"直接决定了技术选择。**

### 关键文件

- `extensions/telegram/src/bot.ts` — Bot 初始化和中间件
- `extensions/telegram/src/bot-updates.ts` — 更新去重和偏移量追踪

---

## 阶段 2：缓冲与去重

### 发生了什么

三道缓冲处理，保证 Agent 收到的是一条完整、不重复的消息：

| 缓冲 | 解决什么问题 | 怎么做 |
|------|------------|--------|
| 去重 | 网络抖动导致 Telegram 重复推送 | 用 update_id 去重，处理过的丢弃 |
| 文本碎片合并 | Telegram 4096 字符限制导致长文被拆分 | 等 1.5 秒，把碎片拼回一条 |
| 防抖 | 用户快速连发多条短消息 | 等用户停止打字，合并成一条 |

### 产品洞察

消息类产品的"接收"远比想象的复杂。不是"收到就处理"，而是要考虑重复、拆分、连发。这些边界情况处理不好，用户感知到的就是"AI 好蠢，我说一句话它回了三次"。

### 关键文件

- `extensions/telegram/src/bot-handlers.buffers.ts` — 媒体组和文本碎片缓冲
- `extensions/telegram/src/bot-handlers.runtime.ts` — 入站防抖

---

## 阶段 3：归一化与权限检查

### 归一化：统一格式

每个渠道的消息格式不同。归一化把 Telegram 特有的格式翻译成内部统一的 MsgContext：

```
Telegram 原始:                    MsgContext:
  message_id: 12345        →       Body: "帮我总结今天的科技新闻"
  from.id: 67890                    From: "67890"
  chat.type: "private"              SenderName: "宋"
  text: "帮我总结..."               SessionKey: "agent:main:telegram:dm:67890"
                                    OriginatingChannel: "telegram"
                                    ChatType: "private"
```

后续所有系统（路由、Agent、记忆）只认 MsgContext，不知道也不关心消息来自哪个渠道。这就是 23 个渠道能统一处理的关键——每个渠道负责把自己的格式"翻译"成 MsgContext。

### 权限检查

```
私聊 → 检查 DM 策略
  配对模式（默认）：需要先用配对码认证
  开放模式：允许任何人
  所有者模式：只有自己能用

群聊 → 检查群组策略
  群在允许列表里吗？
  需要 @ 才回复吗？

未通过 → 消息丢弃，AI 看不到
```

### 关键文件

- `extensions/telegram/src/bot-message-context.ts` — 主上下文构建器
- `extensions/telegram/src/conversation-route.ts` — Agent 路由解析
- `extensions/telegram/src/dm-access.ts` / `group-access.ts` — 权限检查

---

## 阶段 4：路由

模块 2 已经详细讲过。两步走：

```
第一步：匹配绑定规则 → 确定交给 main Agent
第二步：会话范围配置 → 确定 Session Key = agent:main:main
```

### 关键文件

- `src/routing/resolve-route.ts` — 核心路由逻辑
- `src/routing/session-key.ts` — Session Key 构建
- `src/routing/bindings.ts` — 绑定查找

## 阶段 5：上下文组装与 Agent 执行

**整条链路最核心的阶段。** 后续模块 4（Agent 引擎）和模块 7（上下文工程）会分别深入。

### 第一步：上下文组装

在调用模型之前，组装完整的 prompt：

```
System Prompt（你是谁、能做什么、规则）
  + 记忆检索结果（"用户之前说他关注 AI 和半导体"）
  + 对话历史（最近几轮）
  + 当前消息（"帮我总结今天的科技新闻"）
  + 可用工具列表（web_search, read, write...）
  → 打包成完整 prompt
```

### 第二步：执行循环

```
prompt → 调用 Claude API
  ↓
Claude 决定："我需要先搜索"
  → 返回 tool_use: web_search("今天科技新闻")
  ↓
OpenClaw 执行 web_search → 拿到 10 条新闻
  ↓
搜索结果注入对话 → 再次调用 Claude
  ↓
Claude 生成摘要 → 流式输出（边生成边发送）
  ↓
如果 Claude API 挂了？
  → 自动切换备用模型 → 轮换 API Key → 重试
```

### 第三步：记忆更新

对话结束后，提取值得记住的信息存入记忆库，下次相关话题能关联到这次对话。

### 关键文件

- `src/agents/pi-embedded-runner/run.ts` — 主执行循环（重试/切换）
- `src/agents/pi-embedded-runner/run/attempt.ts` — 单次尝试（组装上下文/调模型/执行工具）
- `src/agents/system-prompt.ts` — System Prompt 构建
- `src/memory/search-manager.ts` — 记忆检索

---

## 阶段 6：回复出站

### 发生了什么

```
AI 生成回复文本
  ↓
检查渠道限制（Telegram 4096 字符/条）
  → 超过则智能分块（保留代码块、URL 完整性）
  ↓
流式推送：
  AI 还在生成时，先发一条"草稿消息"
  → 每隔几百毫秒更新内容（用户看到文字在"打字"）
  → 生成完毕后发送最终版本
  ↓
调用 Telegram API sendMessage → 用户看到回复
```

### 为什么要流式推送

等 AI 全部生成完再发 → 用户等 5-10 秒看空白界面。流式推送让用户感觉 AI 在"实时思考"，体验好很多。

### 关键文件

- `extensions/telegram/src/bot-message-dispatch.ts` — 主分发协调器
- `extensions/telegram/src/draft-stream.ts` — 实时预览流
- `src/infra/outbound/deliver.ts` — 通用出站（分块、媒体处理）

---

## 关键设计决策汇总

| 阶段 | 决策 | 原因 |
|------|------|------|
| 到达 | 轮询而非 Webhook | 个人用户没有公网 IP |
| 缓冲 | 三道缓冲（去重/合并/防抖） | 避免 AI 重复回复或处理不完整消息 |
| 归一化 | 统一 MsgContext 格式 | 让后续系统不感知具体渠道 |
| 权限 | 默认配对模式 | 安全优先，陌生人不能随便跟你的 AI 说话 |
| 执行 | 模型-工具循环 + 自动切换 | 让 AI 能调用真实工具 + 保证可用性 |
| 出站 | 流式推送 + 智能分块 | 用户体验（实时反馈）+ 适配渠道限制 |

---

## 产品启发

1. **消息接收的复杂度被低估**：去重、合并、防抖——这些"看不见"的处理直接决定用户对 AI 的第一印象
2. **归一化是多渠道产品的命脉**：一个统一的内部格式，让 N 个渠道的复杂度不会传染到核心系统
3. **模型调用只是全流程的一小部分**：前有上下文组装（决定 AI 看到什么），后有记忆更新（决定 AI 记住什么），中间有工具执行（决定 AI 能做什么）
4. **流式输出是体验的分水岭**：同样的 AI 能力，有没有流式推送，用户感知完全不同
5. **自动切换和重试是生产级的标志**：个人用户不会手动换模型，系统必须自己搞定

---

## 关键术语表

| 术语 | 含义 |
|------|------|
| 长轮询（Long Polling） | 定期向 Telegram 服务器请求新消息的方式 |
| update_id | Telegram 为每条更新分配的唯一 ID，用于去重和偏移量追踪 |
| 防抖（Debounce） | 等待用户停止打字后再合并处理，避免逐条处理碎片消息 |
| MsgContext | OpenClaw 的统一消息格式，所有渠道的消息归一化后都变成这个结构 |
| DM Policy | 私聊策略（pairing/open/owner），控制谁能跟你的 AI 说话 |
| tool_use | 模型返回的工具调用请求，告诉系统"我需要执行某个工具" |
| 流式输出（Streaming） | AI 边生成边发送，用户实时看到文字出现 |
| 草稿消息（Draft） | 流式推送时先发一条不断更新的预览消息 |
| 智能分块（Chunking） | 超长回复按渠道限制拆分，保留代码块和 URL 的完整性 |

# 模块 4：Agent 执行系统

## 一句话总结

Agent 执行的核心是 ReAct 循环（思考→行动→观察→循环），但生产级的可靠性来自外围的三层防护：认证轮换、模型切换、上下文压缩。核心循环人人能写，外围工程决定生死。

---

## 架构地图：四层嵌套循环

```
最外层：模型切换循环（model-fallback.ts）
  Claude 挂了换 GPT，GPT 也挂了换 DeepSeek
  │
  └─ 第二层：认证轮换 + 重试循环（run.ts）
       这个 API Key 限速了换下一个
       最多 160 次迭代（按 Key 数量动态调整）
       │
       └─ 第三层：单次尝试（attempt.ts）
            加载会话 → 注册工具 → 组装 Prompt → 清洗历史 → 调用模型
            │
            └─ 最内层：ReAct 工具循环（核心）
                 模型思考 → 调用工具 → 拿到结果 → 喂回模型 → 循环
                 直到模型决定"够了"生成最终回复
```

---

## 核心：ReAct 循环

```
思考(Think) → 行动(Act) → 观察(Observe) → 思考 → 行动 → 观察 → ... → 最终回复

例：
  用户："帮我总结今天的科技新闻"
  Think: "我需要先搜索新闻"
  Act:   web_search("今天科技新闻")
  Observe: [10 条新闻标题和摘要]
  Think: "信息够了，我来总结"
  → 生成最终回复（流式输出）
```

ReAct 循环本身是公开知识（LangChain/CrewAI/AutoGen 都能实现），OpenClaw 的差异化在于外围的工程可靠性。

---

## 外围：工程可靠性

### 层级总览

| 层 | 核心 vs 外围 | 解决什么 |
|----|------------|---------|
| 工具循环 | **核心（ReAct）** | AI 怎么思考和行动 |
| 重试 + 压缩 | 外围 | 上下文溢出、超时 |
| 认证轮换 | 外围 | API Key 限速/失效 |
| 模型切换 | 外围 | 模型宕机 |
| 流式包装链 | 外围 | 各家 API 差异抹平 |
| 历史清洗 | 外围 | 各家模型的格式怪癖 |

### 模型切换（最外层）

候选列表（如 [Claude, GPT, DeepSeek]）在启动前就构建好。逐个尝试，一个失败通过 `FailoverError` 信号切到下一个。全部失败返回 `FallbackSummaryError`（携带每个模型的失败详情）。

关键区分：用户主动取消（AbortError）会跳过切换（尊重取消意图），超时不会跳过（慢模型应该被替换）。

### 认证轮换（第二层）

解决"同一个模型，不同的钥匙"的问题：

- 迭代预算：基础 24 + 每个 Key ×8，封顶 160
- Key 限速/失效 → 标记冷却期 → 换下一个 Key
- **超时不标记冷却**：超时可能是网络波动不是 Key 的问题，标记会白白浪费好 Key
- 支持 OAuth Token 自动刷新：Token 快过期时后台提前刷新

### 重试策略

| 失败情况 | 处理方式 |
|---------|---------|
| 上下文溢出 | 压缩历史（最多 3 次），再试；还溢出则截断超大工具结果 |
| 超时 + 上下文用了 >65% | 压缩历史（最多 2 次），再试 |
| 认证失败 + 可刷新 Token | 刷新 Token，重试一次 |
| 认证/限速/计费/过载 | 换 Key；Key 用完则抛 FailoverError 到模型切换层 |
| Thinking 级别不支持 | 降级 thinking 级别，重试 |
| 迭代上限 | 返回错误给用户 |

### 上下文压缩 vs 截断

| 方案 | 做法 | 优缺点 |
|------|------|--------|
| 截断 | 删掉最早的对话 | 简单但丢信息 |
| 压缩 | 用 AI 总结之前的对话 | 保留关键信息，减少 token |

OpenClaw 优先压缩（最多 3 次），穷尽后才截断。

---

## 单次尝试的内部流程

### ① 加载会话

打开 JSONL 转录文件 → 获取文件锁（防并发写入）→ 上下文引擎预热

### ② 注册工具

工具工厂创建所有工具 → 8 层策略过滤 → 安全守卫包裹 → 得到"本次可用工具列表"

### ③ 组装 System Prompt

身份 + 可用工具 + 技能 + 记忆指令 + 安全规则 + 工作目录 + 渠道能力 → 完整系统提示词

### ④ 清洗历史消息

- 工具调用和结果不配对 → 修复
- Gemini/Anthropic 格式不兼容 → 转换
- 对话太长 → 按限制截断
- 截断后孤立的工具结果 → 修复

### ⑤ 流式包装链（15 层）

在调用模型 API 之前，请求经过 15 层包装，抹平各家 API 差异：

```
原始 API 调用
  → Ollama 适配（本地模型）
  → OpenAI WebSocket 适配（实时模型）
  → Vertex IAM 认证注入
  → 上下文窗口注入
  → 缓存追踪
  → thinking 标签过滤
  → 工具调用 ID 修复
  → 工具名称规范化
  → 参数 JSON 修复
  → HTML 实体解码
  → 请求日志
```

每个包装层是一个独立的横切关注点，保持核心逻辑不用关心"我在调哪个模型"。

### ⑥ 流式输出

模型边生成边吐 token → 按块缓冲（避免半截 Markdown）→ 过滤 thinking 标签 → 去重 → 回调给投递层 → 推送给用户

---

## 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 模型切换 vs 认证轮换分两层 | 分开 | 认证是"同模型换钥匙"，切换是"换模型降级"，混在一起会导致 Key 限速就换模型 |
| 超时不标记 Key 冷却 | 不标记 | 超时是网络/模型问题不是 Key 问题，标记会浪费好 Key |
| 上下文溢出先压缩后截断 | 压缩优先 | 保留关键信息，最多压缩 3 次才降级到截断 |
| 流式包装链（15 层） | 装饰器链 | 每个提供商的怪癖独立处理，核心逻辑不分支 |
| 迭代上限动态调整 | 按 Key 数量 | Key 多→需要更多轮换机会，但封顶 160 防死循环 |

---

## 产品启发

1. **竞争壁垒不在 ReAct 循环**：核心循环人人能写，外围的认证轮换、模型降级、上下文压缩、多提供商兼容——这些不性感但决定生死
2. **"永远能回复"是生产级的标志**：个人用户不会手动换模型/换 Key，系统必须自动搞定
3. **多提供商兼容的成本**：15 层流式包装链 = 每接一个新模型都有适配工作。这是护城河也是负担
4. **上下文管理是隐藏的核心能力**：压缩 vs 截断的决策直接影响 AI 回复质量

# 模块 5：多 Agent 编排

## 一句话总结

多 Agent 不是为了"更聪明"，而是为了"更高效"和"更安全"。OpenClaw 通过 `sessions_spawn` 工具提供两种派生模式（Subagent 内部派生 / ACP 外部委派），用推送机制（Auto-Announce）返回结果，用深度和并发限制防止失控。

---

## 核心概念

### 为什么需要多 Agent

| 场景 | 单 Agent 的问题 | 多 Agent 的解法 |
|------|----------------|----------------|
| 并行处理 | 只能串行做 3 个调研 | 3 个子 Agent 同时跑 |
| 专业分工 | 写代码和审查代码用同一个权限 | 不同 Agent 不同工具权限 |
| 长时间任务 | 主 Agent 不能一直等 | 后台 Agent 盯着，完成后通知 |

### 派生方式：都是通过工具调用

`sessions_spawn` 是一个普通工具，跟 `web_search` 一样出现在 Agent 的工具列表里。模型决定要派生时就调用它：

```json
{
  "tool": "sessions_spawn",
  "input": {
    "task": "搜索 Mem0 这个产品，分析优劣势",
    "runtime": "subagent",
    "label": "调研 Mem0"
  }
}
```

---

## 两种派生模式

### Subagent（内部派生）

在同一个 Gateway 进程里创建新的会话。不是新进程，是新会话。

```
Gateway 进程（只有一个）
  ├── 主 Agent 会话（agent:main:main）          ← 一直在跑
  ├── 子 Agent A 会话（agent:main:subagent:uuid-a）  ← 新创建
  ├── 子 Agent B 会话（agent:main:subagent:uuid-b）  ← 新创建
  └── 子 Agent C 会话（agent:main:subagent:uuid-c）  ← 新创建
```

类比：不是开了三家分公司（新进程），而是在同一间办公室里开了三个平行会议（新会话）。

每个子 Agent：
- 有独立的对话历史（互不干扰）
- 有自己的 System Prompt（标注为子 Agent，受限角色）
- 共享工具集（但权限可能更受限）
- 共享工作目录
- 自己也在跑完整的 ReAct 循环

### ACP（外部委派）

通过网络协议连接到外部 AI 系统（Claude Code、Codex 等）。也是同一个 `sessions_spawn` 工具，只是 `runtime` 参数不同。

```json
{
  "tool": "sessions_spawn",
  "input": {
    "task": "用 Python 写一个记忆层 MVP",
    "runtime": "acp",
    "streamTo": "parent"
  }
}
```

### 对比

| 维度 | Subagent（内部） | ACP（外部） |
|------|-----------------|------------|
| 在哪跑 | 同一个 Gateway 进程 | 外部独立进程 |
| 共享什么 | 工作目录、工具系统 | 什么都不共享，只通过协议通信 |
| 能传文件 | 能（最多 50 个） | 不能 |
| 能指定模型 | 能 | 不能 |
| 能恢复已有会话 | 不能 | 能（resumeSessionId） |
| 能实时看进度 | 不能（等完成后通知） | 能（streamTo="parent"） |
| 沙箱内能用 | 能 | 不能（跑在宿主机上） |

什么时候用哪个：
- 并行搜索/分析 → Subagent
- 让 Claude Code 写代码 → ACP
- 后台盯着某个事情 → Subagent + 线程绑定

---

## 结果回传：推送机制（Auto-Announce）

### 核心设计：推送而不是轮询

```
❌ 轮询：父 Agent 每隔几秒问"完成了吗？" → 浪费 token
✅ 推送：子 Agent 完成后主动把结果注入到父 Agent 的对话中
```

类比：你给下属布置任务后不需要每五分钟问一次，下属做完了会主动来汇报。

### 推送的具体实现

父 Agent 派完任务后就停着，不做任何事。子 Agent 完成后：

```
子 Agent 完成
  ↓ 冻结结果（保存到磁盘，防丢失）
  ↓ 检查有没有自己的子孙还在跑（有就等）
  ↓ 向父 Agent 的对话注入一条系统消息：
    "✅ 子任务'调研 Mem0'已完成，结果：……"
  ↓ 父 Agent 正在忙？排队等空闲再投递
  ↓ 投递失败？重试 3 次（1s/2s/4s 间隔）
  ↓ Gateway 崩了？重启后从磁盘恢复，继续投递
```

父 Agent 收到注入的消息后被"唤醒"，就像有人给它发了一条新消息。收齐所有子 Agent 的结果后，汇总生成最终回复。

### 为什么选推送

- 不浪费 token（轮询每次都要调模型）
- 结果持久化到磁盘（Gateway 重启不丢失）
- 支持多层嵌套（子 Agent 的子 Agent 完成后逐层汇报）

---

## 深度与并发控制

### 层级角色

```
深度 0（主 Agent）→ 角色 main，可以派生
深度 1（子 Agent）→ 角色 orchestrator（可继续派生）或 leaf（不能派生）
深度 2（孙 Agent）→ 角色 leaf，不能再派生
超过 maxSpawnDepth → 拒绝
```

### 并发限制

| 限制 | 默认值 | 作用 |
|------|--------|------|
| maxSpawnDepth | 可配置 | 最多几层嵌套 |
| maxChildrenPerAgent | 5 | 一个 Agent 同时最多跑几个子 Agent |
| runTimeoutSeconds | 可配置 | 子 Agent 最长跑多久 |
| allowAgents | 同 Agent | 允许派生哪些 Agent 身份 |

为什么严格限制：每个子 Agent 都在调 API、消耗 token。递归失控可能产生几百美元账单。

---

## 线程绑定

### 什么是"线程"

不是编程概念，是聊天软件的功能。Discord/Slack/Telegram 里，回复某条消息会创建一个"话题"——附属在主消息下面的子对话。

### 线程绑定的效果

```
Discord 频道 #general

  你：帮我分析这个项目的代码
  AI：任务比较复杂，我创建了一个专属线程来处理。
  └── 📎 线程："代码分析任务"    ← 点击展开
       ├── 子 Agent：开始分析项目结构……
       ├── 子 Agent：发现了 3 个主要模块……
       ├── 你：重点看看安全模块          ← 你可以追问
       ├── 子 Agent：好的，安全模块是……
       └── 子 Agent：分析完成，报告如下……
```

不用线程绑定 → 子 Agent 后台默默干活，你只能等最终结果
用了线程绑定 → 子 Agent 在一个可以点进去的子对话框里干活，你随时可以看进度、追问、修改要求

### 怎么触发

```json
{
  "tool": "sessions_spawn",
  "input": {
    "task": "分析项目代码结构",
    "runtime": "subagent",
    "thread": true,
    "mode": "session",
    "label": "代码分析任务"
  }
}
```

`thread: true` → 创建聊天线程
`mode: "session"` → 子 Agent 会话持续存在，可以继续互动

## 产品启发

1. **推送 vs 轮询揭示了 AI 产品的成本取舍**：轮询简单但每次都要调模型浪费钱，推送复杂（1700 行代码）但高效。长期运行的 AI 产品，效率比简单性更重要。

2. **深度限制是产品安全设计**：不是技术限制，是防止 AI 给自己制造无限任务烧光 API 预算。给 AI 的自主权必须有边界。

3. **两种派生对应两种商业场景**：Subagent = 自建团队（快速、可控），ACP = 外包（专业、隔离）。做 AI 平台时分别对应内部编排和第三方接入。

4. **线程绑定把异步任务变成交互式工作空间**：用户感觉"AI 在帮我做事，我随时可以插手"，而不是"任务扔出去了不知道什么时候回来"。这是产品体验的巨大差异。

5. **同一个工具、不同的 runtime**：`sessions_spawn` 一个工具入口覆盖两种模式，对模型来说学习成本低。对产品设计的启发：能力多样性应该隐藏在参数后面，而不是暴露成多个工具。

# 模块 6：记忆系统

## 一句话总结

OpenClaw 的记忆系统把用户写的 Markdown 文件或对话中自动提取的事实，切块后同时建立向量索引和全文索引，通过混合搜索（70% 语义 + 30% 关键词）找到相关记忆，再以"先广搜摘要、再精取全文"的两步检索注入到 AI 的上下文中。向量仅用于检索，AI 看到的始终是原文。

---

## 两种记忆哲学

| | memory-core（用户控制，生产默认） | memory-lancedb（自动提取，实验性） |
|---|---|---|
| 谁决定记什么 | 用户手动写 MEMORY.md 等文件 | AI 根据硬编码规则自动判断 |
| 判断逻辑 | 不涉及（用户写的就是确定的） | 正则规则匹配（决策类、偏好类、个人信息类关键词），非 LLM 判断 |
| 存储格式 | Markdown 文件 + SQLite（原文+向量+全文索引） | LanceDB（原文+向量，无全文索引） |
| 搜索方式 | 向量 + 全文混合搜索 | 纯向量搜索 |
| 怎么注入 | AI 主动调 memory_search/memory_get 工具 | 每轮对话前自动注入 |
| 每次对话最多提取 | N/A | 3 条，去重（余弦相似度 > 0.95 则跳过） |
| 可靠性 | 高（用户写的就是确定的） | 中（规则可能漏判或误判） |

### 第一个产品决策："谁来决定记什么"

- 用户控制：可靠但需要用户主动维护文件
- 自动提取：方便但可能记错（"考虑用 PostgreSQL"被记成"已决定用 PostgreSQL"）
- 可能的折中：AI 自动提取候选记忆，用户确认后才生效

---

## 存储架构

### 存什么

不管哪种模式，**文本和向量都同时存储**。向量仅用于检索索引，AI 看到的始终是原文。

### 一个文本块的三种存在形式（memory-core）

```
chunks 表：存原文（source of truth）
  → text, path, start_line, end_line, hash

chunks_vec 表：存向量（语义搜索用的索引）
  → id, embedding FLOAT[N]

chunks_fts 表：存全文索引（关键词搜索用的索引）
  → text（BM25 可搜索）

三者通过同一个 chunk ID 关联
搜索时各走各的路，合并时认同一个 ID
```

### 为什么用 SQLite

目标用户在本地跑。SQLite 零依赖、不需要启动服务、文件复制就能备份。sqlite-vec 扩展添加近似最近邻搜索。如果扩展加载失败（某些平台），降级到 JavaScript 暴力计算余弦相似度。

### 切块策略

| 参数 | 默认值 | 原因 |
|------|--------|------|
| 块大小 | 400 token（约 1600 字符） | 太大→嵌入信号被稀释；太小→丢失上下文 |
| 重叠 | 80 token（块大小的 20%） | 防止切断跨边界的完整概念 |

算法：逐行累积，超过上限就刷出一个块；新块继承上一块末尾的重叠部分。

### 嵌入提供商

| 提供商 | 默认模型 | 特点 |
|--------|---------|------|
| OpenAI | text-embedding-3-small | 远程，最常用 |
| Gemini | text-embedding-004 | 远程，支持自定义维度 |
| Voyage | voyage-3-lite | 远程，有异步批处理 |
| Mistral | mistral-embed | 远程 |
| Ollama | nomic-embed-text | 本地服务 |
| local | embeddinggemma-300m | 完全离线（node-llama-cpp） |

自动选择逻辑：有本地模型文件→先试本地 → 再试远程（openai → gemini → voyage → mistral）→ 都没 key → 退化为纯关键词搜索。

### 嵌入缓存

缓存 key = 内容哈希 + 提供商 + 模型 + 端点哈希。

- 文件内容没变 → 内容哈希不变 → 不重新调 API → 省钱
- 换了嵌入模型 → 模型字段变了 → 缓存自动失效 → 不会用旧向量配新查询
- 换了端点（两个不同的 OpenAI 兼容服务） → 端点哈希变了 → 缓存失效

---

## 混合搜索

### 并行执行两种搜索

同一批切块同时有两个索引。搜索时并行走两条路：

```
查询："数据库用的什么？"
     ├──→ 向量搜索：嵌入查询 → 余弦相似度排序
     └──→ 关键词搜索：提取关键词 → BM25 匹配排序
```

### 加权合并

每个块只出现一次，分数是两种搜索的加权和：

```
最终分数 = 0.7 × 向量分 + 0.3 × 关键词分
```

- 两边都找到的块：分数最高（两票都投了）
- 只有一边找到的块：分数被拉低（只有一票）
- 低于阈值（默认 0.35）的块被过滤掉

### 为什么 70/30

语义搜索通常更可靠（理解同义词和意图），但关键词搜索在专有名词、版本号、人名等精确匹配上不可替代。比例可配置。

### 特殊情况处理

当 textWeight=0.3 且 minScore=0.35 时，纯关键词结果（向量分=0，最高分=0.3）永远过不了阈值。系统检测到这种情况后，对出现在关键词结果中的块放宽阈值，避免有用结果被误杀。

---

## 两步检索

### 为什么分两步

一步返回全部内容 → 6 个文件可能占几千 token → 浪费上下文空间
两步走 → 先看摘要判断相关性 → 再精确获取需要的部分 → 节省 token

### memory_search（广搜）

输入：查询文本 + 可选的 maxResults、minScore
返回：每条结果包含 path、startLine、endLine、score、snippet（最多 700 字符）、citation

AI 用摘要判断哪条相关。

### memory_get（精取）

输入：文件路径 + 可选的 from（起始行）、lines（行数）
返回：指定范围的原始文件内容

AI 只获取真正需要的段落。

### System Prompt 中的强制指令

```
"Before answering anything about prior work, decisions, dates, people,
 preferences, or todos: run memory_search first, then use memory_get
 to pull only the needed lines."
```

这是命令而非建议——AI 在回答特定类型问题前必须先查记忆。

---

## 会话记忆：对话 → 可搜索记忆

对话历史（JSONL 文件）可以作为记忆源（需配置开启）：

```
JSONL 对话记录 → 提取纯文本（脱敏）→ 切块 → 嵌入 → 存入同一个 SQLite
搜索时跟 MEMORY.md 的块混在一起排序
```

### 增量同步

不是每句话都重新嵌入。Delta 追踪：新增内容超过 100KB 时才触发重新索引。在"记忆新鲜度"和"API 成本"之间权衡。

### 同步触发点

1. 文件变更（chokidar 监听，1.5s 防抖）
2. 会话增量（5s 防抖 + 100KB 阈值）
3. 会话开始时（可配置）
4. 定时后台同步
5. 搜索时（如果有脏数据，非阻塞）

---

## 高级功能（默认关闭）

### MMR（最大边际相关性）

防止返回结果太相似。如果记忆里有 5 段都讲数据库选型，纯按相关性排会返回 5 段重复内容。MMR 在相关性和多样性之间平衡：

```
lambda = 0.7 → 70% 看相关性，30% 看与已选结果的差异性
```

用 Jaccard 相似度（词集合交集/并集）而非余弦相似度来计算多样性，计算成本更低。

### 时间衰减

指数衰减：越老的记忆分数越低。半衰期默认 30 天。

精妙设计：**只对有日期的文件做衰减**（如 `memory/2024-01-15.md`）。永久性的 `MEMORY.md` 和无日期的主题文件不衰减。日记类临时记忆自然淡化，刻意记录的决策永远全权重。

```
用户写 MEMORY.md（或 AI 自动提取事实）
  ↓
切块（400 token/块，80 token 重叠）
  ↓
嵌入（调用嵌入 API，缓存命中则跳过）
  ↓

```

## 产品启发

1. **"谁来决定记什么"是第一个产品决策**：用户控制可靠但麻烦，自动提取方便但易错。折中方案：AI 提取候选 + 用户确认。

2. **混合搜索是必须的**：纯向量搜索在精确匹配上失败，纯关键词搜索在语义理解上失败。做记忆产品必须两者结合。

3. **两步检索是 token 成本控制的关键**：提供"先搜摘要、再取全文"的 API 设计，让调用方控制预算。

4. **SQLite 做向量存储被低估了**：个人/小团队场景零运维成本，对 MVP 来说完全够用。

5. **嵌入缓存的 key 设计值得复用**：内容哈希 + 提供商 + 模型 + 端点哈希——换模型自动失效，同内容不重复调 API。

6. **时间衰减只对"有日期的记忆"生效**：区分"永久知识"和"临时记录"，让重要决策不会随时间淡化。

# 模块 7：上下文工程

## 一句话总结

上下文工程解决的是"AI 的视野有限但信息不断增长"的问题。通过精心组装 System Prompt、按需注入记忆、四层防溢出（截断→裁剪→压缩→恢复）、以及多模型格式清洗，确保 AI 每次都在有限的上下文窗口里看到最有价值的信息。

---

## 核心问题

AI 模型的上下文窗口是固定的（Claude 200K、GPT 128K），但对话会不断增长。上下文工程就是在有限的窗口里放最有价值的信息——像给新同事做 30 分钟工作交接，不是念三个月的会议纪要，而是挑最重要的说。

---

## 上下文的组成

| 组成部分 | 是否一定有 | 大小 |
|---------|-----------|------|
| System Prompt（身份+工具+规则） | 一定有 | 2000-4000 token |
| 项目文件（SOUL.md/AGENTS.md 等） | 看用户有没有创建 | 几百到几千 token |
| 记忆检索结果 | **不一定**——AI 主动调 memory_search 才有 | 几百 token |
| 对话历史 | 一定有（非首轮） | 占大部分空间 |
| 当前用户消息 | 一定有 | 几十到几百 token |
| 可用工具列表 | 一定有 | 几百 token |

记忆不是自动注入的。是 AI 根据 System Prompt 里的指令，判断当前问题需要查记忆时，主动调用 memory_search 工具。如果用户只是说"你好"，AI 不会查记忆，这次上下文里就没有记忆内容。

---

## System Prompt：AI 的身份证 + 操作手册

### 25 个组成部分（full 模式）

最重要的（必有）：
1. 身份声明
2. 可用工具列表
3. 安全规则
4. 工作目录
5. 授权发送者（可 HMAC 哈希保护隐私）

按需注入的：
6. 技能列表
7. 记忆使用指令
8. 沙箱信息
9. 消息路由
10. 语音/TTS 设置
11. ...更多

用户自定义的项目上下文：
- SOUL.md — AI 的人格和语气
- AGENTS.md — 操作指令
- USER.md — 用户画像

### 三种模式

| 模式 | 用于 | 大小 |
|------|------|------|
| full | 主 Agent（用户直接聊） | 2000-4000+ token |
| minimal | 子 Agent / 定时任务 | 约 full 的 1/3 |
| none | 极简场景 | ~20 token |

子 Agent 用 minimal：一个被派去搜索竞品的子 Agent 不需要消息路由规则、emoji 设置、语音选项。省 token，减少干扰。5 个并行子 Agent 用 full = 20000 token 浪费在系统指令上，用 minimal 可省 2/3。

---

## 四层防溢出

**不是每次都全部执行。** 按需触发，像水位线——水位低什么都不做，涨高了才逐层启动。

### 第 1 层：历史截断（配置了上限才执行）

```
配了"DM 最多 20 轮"→ 超过 20 轮时截断前面的
没配 → 保留全部历史
可针对每个渠道、每个用户单独配置
```

截断后可能导致工具调用配对断裂（tool_use 被截掉但 tool_result 还在），系统会自动修复。

### 第 2 层：工具结果裁剪（轻量级，不调模型）

```
5 分钟内的工具结果 → 保留原文
超过 5 分钟且超过 4000 字 → 保留前 1500 字 + 后 1500 字 + "[已裁剪]"
如果还不够 → 整个替换成"[旧的工具结果已清除]"
```

关键：只影响本次发给模型的内容，磁盘上的原始记录不变。

### 第 3 层：自动压缩（重量级，调模型做总结）

```
触发条件：对话历史占上下文窗口 > 65%
执行：让模型生成对话摘要 → 用摘要替换旧对话
效果：50 轮对话 80000 token → 摘要 2000 token + 最近几轮 8000 token
代价：需要一次模型调用（消耗 token 和时间）
```

压缩前会检查是否有真实对话内容——如果历史里只有心跳和系统事件，跳过压缩不浪费 token。

### 第 4 层：溢出恢复（兜底）

```
模型返回"上下文溢出"错误 → 立即触发压缩（最多 3 次）
压缩 3 次还溢出 → 截断超大工具结果 → 重试
```

### 触发关系

```
短对话（5 轮以内）→ 四层一个都不触发，完整历史直接发
中等对话（20+ 轮）→ 可能触发截断
长对话（历史占 65%+）→ 触发压缩
异常情况 → 溢出恢复兜底
```

---

## 消息清洗：多模型格式兼容

每次发给模型前必须执行（不是按需，是必须），因为不同模型对消息格式要求不同：

| 模型 | 要求 | 清洗措施 |
|------|------|---------|
| Claude | user/assistant 严格交替 | 连续 user 消息合并 |
| Gemini | 不能以 assistant 开头 | 补虚拟 user 消息 |
| GPT | 工具 ID 和推理块有特殊格式 | 修复 ID、删不完整的推理块 |
| 所有 | tool_use 必须配对 tool_result | 自动修复配对、补合成错误结果 |
| 所有 | 压缩前的 token 统计会导致重复计算 | 旧统计归零 |

**为什么需要：** 用户可以中途换模型。Claude 格式的历史可能不符合 GPT 的要求。清洗管道抹平差异。

### 安全清洗

System Prompt 中嵌入的路径等文本会被清除 Unicode 控制字符——防止目录名包含换行符时注入 prompt 指令。

---

## Token 预算管理

### 怎么知道模型的上下文窗口有多大

查找顺序（优先级从高到低）：
1. 用户手动指定的覆写值
2. 配置文件中的值
3. 模型自己报告的值（API 发现）
4. 模型目录缓存
5. 默认值（128K）

### 安全策略

- 同一模型在多个提供商下窗口不同 → 取最小值（宁可早压缩不要溢出）
- 窗口低于 16K → 拒绝运行
- 窗口低于 32K → 警告用户
- 可以手动设上限（即使模型支持 1M，也可以限制在 200K 控制成本）

## Context Engine 插件系统

OpenClaw 预留了一个插件接口，允许第三方完全替换上下文组装策略：

| 生命周期方法 | 用途 |
|------------|------|
| bootstrap() | 会话初始化时导入历史上下文 |
| ingest() | 每条消息到达时索引/存储 |
| assemble() | **核心**——模型调用前返回应该看到的消息列表 |
| compact() | token 压力时压缩上下文 |
| afterTurn() | 回复后持久化、触发后台压缩 |
| dispose() | 释放资源 |

默认的 LegacyContextEngine 是空操作（委托给现有代码）。这意味着加自定义引擎是纯加性的——比如未来可以接一个 RAG 引擎，只返回最相关的历史轮次而不是完整时间线。

```

  ↓
构建 System Prompt（身份+工具+规则+人设）
  ↓
加载对话历史
  → 消息清洗（每次必做，修复格式）

  ↓

```

## 产品启发

1. **上下文工程是 AI 产品最被低估的核心能力**：用户觉得"AI 记得之前说的话"不是模型的功劳，是上下文工程精心组装的结果

2. **四层防溢出 = 用户永远不碰到错误**：截断→裁剪→压缩→恢复，逐层升级。大部分 AI 产品只有截断，对话一长就丢信息

3. **消息清洗是多模型支持的隐性成本**：每加一个模型就要处理它的格式怪癖。如果只支持一个模型可以省掉这部分

4. **子 Agent 用 minimal prompt 直接影响成本**：5 个并行子 Agent 省 2/3 系统指令 token

5. **token 预算策略要保守**：宁可早压缩（损失一点信息），不要溢出（浪费整次调用的 token）

# 模块 8：插件 SDK 与扩展生态

## 一句话总结

插件是一个"注册行为"而非"运行中的实体"——它在启动时把工具、Prompt 片段、钩子、渠道适配器等能力注册到系统各处，注册完就消失了。82 个扩展通过统一的 SDK 合约和两层加载机制（JSON 身份证 + 按需加载代码）实现即插即用。

---

## 核心概念

### 插件的本质

插件 = 工具 + System Prompt 注入 + 钩子 + 其他能力的打包注册。它本身不是实体，是一个搬运工——把各种能力搬到该去的地方，搬完就走。

系统运行时没有"Brave 插件在跑"的概念，只有"web_search 工具的 Brave 实现在被调用"。AI 也看不到插件，只看到工具列表。

### 一个插件能包含的全部内容

| 内容 | 作用 | 是否必须 |
|------|------|---------|
| 工具（Tools） | 给 AI 新能力 | 可选 |
| System Prompt 注入 | 告诉 AI 怎么用新能力 | 可选 |
| 生命周期钩子（Hooks） | 在系统各环节插入逻辑 | 可选 |
| 渠道适配器（Channel） | 接入新的消息平台 | 可选 |
| 模型提供商（Provider） | 接入新的 AI 模型 | 可选 |
| CLI 命令 | 给命令行加功能 | 可选 |
| HTTP 路由 | 给 Gateway 加 API | 可选 |
| 后台服务（Service） | 持续运行的后台任务 | 可选 |
| 配置 Schema | 定义需要什么配置 | 一般都有 |

---

## 四种插件类型

| 类型 | 数量 | 需要实现什么 | 复杂度 |
|------|------|------------|--------|
| 工具插件 | 十几个 | 一个函数：输入→执行→返回 | 最简单（10 行） |
| 提供商插件 | 30+ | 模型列表、认证方式 | 中等 |
| 记忆插件 | 2 个 | 独占槽位，工具+Prompt+索引 | 中等 |
| 渠道插件 | 23 个 | 最多 20 个适配器 | 最复杂（几千行） |

## 真实插件对比

### Brave（最简单）

```typescript
register(api) {
  api.registerWebSearchProvider(createBraveWebSearchProvider());
}
```

包含：搜索提供商 × 1 + 配置。10 行代码。

### memory-core（中等）

包含：工具 × 2（memory_search、memory_get）+ System Prompt 注入 × 1 + 钩子 × 1（session_start 预热）+ CLI × 1 + 独占槽位声明 + 配置。

### Telegram（最复杂）

包含：渠道适配器十几个（config/gateway/outbound/setup/status/security/pairing/messaging/threading/actions）+ 后台服务 × 1（长轮询）+ 配置。几千行代码。

## 插件加载机制：两层设计

### 第 1 层：只读 JSON 身份证（毫秒级，不执行代码）

读取 `openclaw.plugin.json` 获取元数据：ID、配置 Schema、认证环境变量、声明的渠道/提供商。

用途：启动状态显示、引导安装选择列表、认证流程——都不需要加载插件代码。

### 第 2 层：加载 TypeScript 代码（按需）

只对已启用的插件执行。用 jiti 加载 TypeScript 源码，不需要预编译。

三种加载模式：
- full：所有适配器（正常运行）
- setup-only：只加载配置适配器（引导安装）
- setup-runtime：最轻量（未配置的渠道）

**启动时间只跟"用了几个插件"有关，不跟"支持几个插件"有关。**

### 扫描四个目录

1. 配置文件指定的路径
2. 项目工作目录 `.openclaw/extensions/`
3. OpenClaw 内置 `dist/extensions/`（82 个）
4. 全局 `~/.openclaw/extensions/`（第三方安装）

---

## 插件隔离

| 层面 | 隔离方式 |
|------|---------|
| 代码 | 只能从 `openclaw/plugin-sdk/*` 导入，不能访问核心代码或其他插件代码 |
| 运行时 | 每个插件独立的 Runtime Proxy，子 Agent 操作互不影响 |
| 安全检查 | 非内置插件检查：符号链接逃逸、文件权限、所有者、允许列表 |
| 进程 | 无进程级隔离（所有插件同一进程），有意的取舍——个人场景够用 |

---

## 28 个生命周期钩子

插件可以在系统运行的各个环节插入逻辑：

| 阶段 | 钩子 |
|------|------|
| 消息相关 | message_received, message_sending, message_sent |
| AI 相关 | before_prompt_build, before_agent_start, agent_end, llm_input, llm_output |
| 工具相关 | before_tool_call, after_tool_call |
| 会话相关 | session_start, session_end, before_compaction, after_compaction |
| 多 Agent | subagent_spawning, subagent_spawned, subagent_ended |
| 系统 | gateway_start, gateway_stop |

安全门控：`before_prompt_build` 等能注入 Prompt 的钩子可以被配置禁用，防止不受信任的插件偷改 AI 行为。

---

## 运行时行为

- 插件加载发生在 Gateway 启动时，不支持运行中热加载
- 装了新插件需要重启 Gateway（几秒，守护进程自动管理）
- 配置变更（路由、模型选择等）支持热更新，不需要重启
- 记忆插件独占槽位：同一时间只能有一个活跃

## 外部插件安装

```
用户安装：openclaw plugins install <npm-package>
  → 下载到 ~/.openclaw/extensions/<id>/
  → 执行 npm install --omit=dev
  → 写入配置 config.plugins.installs.<id>
  → 重启 Gateway 后生效
```

重复处理：内置和第三方同 ID 插件共存时，按优先级选（配置 > 全局安装 > 内置 > 工作目录 > 全局自动）。可以用第三方版本覆盖内置版本。

## 产品启发

1. **两层加载是平台型产品的关键**：不加载代码就能知道插件是什么，让 UI、搜索、安装推荐都不依赖代码执行

2. **最简单的插件 10 行代码**：这是生态繁荣的前提。SDK 设计要克制。

3. **四种类型覆盖 AI 产品全部扩展点**：渠道（用户在哪）、提供商（用哪个 AI）、工具（能做什么）、记忆（记住什么）

4. **独占槽位解决"只能有一个"**：记忆后端只能有一个活跃，用配置选择而非代码处理冲突

5. **TypeScript 直接加载**：不需要编译步骤，降低开发者门槛，对生态早期增长重要

6. **钩子的安全门控**：Prompt 注入类钩子可以被禁用——信任模型是"默认不信任第三方"

# 模块 9：渠道抽象与路由

## 一句话总结

23+ 渠道通过"双向翻译层"统一接入：入站时各渠道把自己的消息格式翻译成标准 MsgContext，出站时把通用回复翻译回各平台原生格式。核心系统只认一种格式，完全不感知渠道差异。

---

## 核心设计：翻译层

```
Telegram 格式 ──┐                              ┌── Telegram 原生回复
WhatsApp 格式 ──┤→ 统一 MsgContext → 核心系统 →├── WhatsApp 原生回复
Slack 格式 ────┤   （路由/Agent/记忆）          ├── Slack Block Kit
Discord 格式 ──┘                              └── Discord Embed
```

每个渠道插件负责双向翻译，核心系统永远不需要知道各平台的特有概念。

---

## 渠道适配器：可选插槽设计

每个渠道最多可实现 20 个适配器，但只需实现自己支持的部分：

| 类别 | 适配器 | 必须？ |
|------|--------|--------|
| 基础 | config（配置）、gateway（连接）、outbound（发消息） | 必须 |
| 常见 | setup（安装引导）、status（健康检查）、security（权限） | 常见 |
| 高级 | threading（线程）、actions（反应/编辑）、pairing（配对） | 看平台 |

IRC 没有反应 emoji 就不实现 actions，Telegram 没有 Block Kit 就不实现富文本——可选插槽避免了大量空实现。

---

## 多账号支持

一个渠道类型可以有多个账号，各自独立：

```
渠道 telegram:
  ├── @work_bot → 绑定 work Agent
  └── @life_bot → 绑定 life Agent
```

每个账号独立连接、独立健康状态、独立认证。

---

## 能力声明

渠道声明自己支持什么，核心系统动态适配：

- 不支持图片 → System Prompt 不提"发送图片"
- 支持线程 → 子 Agent 线程绑定可用
- 最大消息长度不同 → 出站分块策略不同

新增渠道不需要改核心代码。

---

## 产品启发

1. **先定义统一消息格式（MsgContext），再写渠道适配**——这是多渠道产品的第一步
2. **可选适配器比统一接口更务实**——每个渠道只做自己能做的
3. **能力声明让核心系统自动适配**——不是硬编码"if telegram then..."
4. **多账号设计从一开始就预留**——个人用一个，团队用多个，不需要重构

# 模块 10：安全模型

## 一句话总结

OpenClaw 的安全模型分三层：谁能说话（访问控制）、AI 能做什么（工具权限）、在哪里做（沙箱隔离）。哲学是"安全默认 + 显式开关"——默认锁死一切，需要时显式打开。

---

## 核心问题

OpenClaw 跑在用户自己的电脑上，AI 能执行代码、控制浏览器、读写文件。跟 ChatGPT 不同（出事是 OpenAI 的事），这里出事是用户自己的事。安全是"没有就不能用"。

---

## 三层安全模型

### 第 1 层：谁能说话——访问控制

| DM 策略 | 行为 | 安全级别 |
|---------|------|---------|
| pairing（默认） | 陌生人需要配对码认证才能聊 | 高 |
| owner | 只有自己能用 | 最高 |
| open | 任何人都能用（需显式开启） | 低 |

群组默认不响应，需要加入允许列表，可配置必须 @ 才回复。

为什么默认 pairing：AI 有执行代码能力，不控制入口等于开放远程代码执行。

### 第 2 层：AI 能做什么——工具权限

**工具配置文件（4 级）：**

| 级别 | 能力 |
|------|------|
| minimal | 只能看状态 |
| coding | 读写文件、执行命令、搜索 |
| messaging | 发消息、管理会话 |
| full | 所有工具 |

每个 Agent 可独立配置，在此基础上还能单独允许/禁止特定工具。

**安全二进制白名单：** AI 执行命令时，不在白名单的命令需要用户手动批准。常见开发工具（ls/git/node）允许，危险命令（rm -rf/dd）拦截。

**SSRF 防护：** AI 访问 URL 时，内网地址（127.0.0.1/192.168.x.x/10.x.x.x）被拒绝，防止被诱导访问内网服务。

### 第 3 层：在哪里做——沙箱隔离

| 模式 | 隔离程度 | 适合场景 |
|------|---------|---------|
| Docker 容器 | 最高（独立文件系统，销毁即还原） | 处理不信任的代码 |
| systemd-run | 中（Linux 进程隔离，限制资源） | Linux 轻量隔离 |
| 原生 | 无（直接在主机操作） | 完全信任 AI 的场景 |

三种模式自动降级：Docker 不可用 → 试 systemd → 原生。可按 Agent 单独配置。

---

## 文件系统边界

即使原生模式，AI 也只能在工作目录（~/.openclaw/workspace/）内读写。每个文件工具执行前检查路径是否在允许范围内。

---

## 安全审计

`openclaw doctor` 一键扫描所有配置，生成三级报告：
- info：一切正常
- warn：有改进空间
- critical：必须修复

---

## 凭证保护

- 加密存储（~/.openclaw/credentials/）
- 日志自动脱敏
- 支持环境变量引用
- Gateway Token 认证
- 多 Key 自动轮换

---

## 三层对应三种威胁

```
第 1 层（谁能说话）→ 防未授权的人使用 AI
第 2 层（能做什么）→ 防 AI 做超出预期的事
第 3 层（在哪里做）→ 即使做了坏事也限制影响范围
```

---

## 产品启发

1. **"安全默认 + 显式开关"**：默认锁死，需要时显式打开。不是出了问题再加限制。
2. **安全不是一个功能，是每个功能的属性**：工具有权限、命令有白名单、文件有路径守卫、网络有 SSRF 防护、凭证有脱敏。
3. **`openclaw doctor` 是产品创新**：安全体检命令化，用户不需要懂安全就能发现问题。
4. **三层是最小安全模型**：缺任何一层都有漏洞。做 AI 产品必须覆盖这三层。

# 模块 11：增长策略与创业机会

## 一句话总结

OpenClaw 的增长靠三个引擎：真实未满足需求（跨渠道 AI 助手）、开源飞轮（社区贡献渠道/提供商）、技术壁垒（5200 文件的真实复杂度）。创业机会在垂直 Agent、记忆中间件、安全审计、插件市场和生态服务五个方向。

---

## OpenClaw 为什么能火

### 引擎 1：真实的未被满足的需求

用户想要：一个了解我、能帮我做事、在所有聊天软件里都能找到的 AI 助手。

- ChatGPT → 只在自己界面用，不了解你的文件
- 各平台 Bot → 各自独立，没有共享记忆
- 自己写 → 门槛太高

OpenClaw 是第一个全部打包好的开源方案。

### 引擎 2：开源 + 插件 = 增长飞轮

```
开源 → 开发者免费用 → 贡献新渠道/提供商
  → 覆盖更多平台 → 更多用户 → 更多贡献者 → 飞轮
```

82 个插件中很多是社区贡献（Zalo=越南、飞书=中国、LINE=日本），团队不需要懂每个平台。

### 引擎 3：技术壁垒足够深

5200 个文件不是臃肿，是真实复杂度：23 渠道、30+ 模型、多 Agent、记忆、上下文工程、安全。后来者很难追赶。

---

## 五个创业机会

### 机会 1：垂直场景 AI Agent 产品

锁定一个垂直场景，不需要通用：

| 场景 | 只需接入 | 记忆优化方向 |
|------|---------|------------|
| 跨境电商客服 | WhatsApp + Telegram + 飞书 | 客户信息、订单历史 |
| AI 编程助手 | IDE + CLI | 项目上下文、代码偏好 |
| AI 教育导师 | 微信 + Web | 学习进度、知识掌握 |
| AI 家庭助手 | WhatsApp + 语音 | 家庭成员偏好、日程 |

好处：不需要 23 渠道（3 个够）、可以预装配置（不需要用户搭）、可以做 SaaS。

### 机会 2：AI 记忆层中间件

把模块 6 学到的记忆系统做成独立 API 服务：

```
POST /memory/store → 存入记忆
POST /memory/search → 混合搜索（向量+关键词）
GET  /memory/get → 精确获取
通过 MCP 协议暴露 → 任何 AI 应用都能接入
```

核心技术点（全部在模块 6 学过）：
- 切块（400 token，80 重叠）
- 混合搜索（70% 向量 + 30% 关键词）
- 两步检索
- 嵌入缓存
- 增量同步 + 时间衰减

目标客户：做垂直 AI 产品的团队（需要记忆但不想自建 RAG）。

### 机会 3：AI Agent 安全审计 SaaS

扫描任何 AI Agent 配置，检测安全问题：
- 工具权限过大
- 缺少 SSRF 防护
- 没有沙箱
- 凭证暴露

类似代码安全扫描（Snyk/SonarQube），但针对 AI Agent。市场正在形成，几乎没有竞争者。

### 机会 4：插件/技能市场

在 ClawHub 基础上扩大：付费插件、模板市场、开发者工具链。类似 Shopify App Store。

### 机会 5：围绕生态的服务

- 托管部署（¥99/月一键部署）
- 企业咨询
- 插件定制开发
- 培训课程
