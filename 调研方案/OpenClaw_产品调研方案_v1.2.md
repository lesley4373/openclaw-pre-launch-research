## Part 0：基准清单整理

> **性质**：桌面研究（desk research），不需要访谈或实测，1–3 天可完成。
>
> **目的**：在启动任何实测或访谈之前，先建立两份基准清单——调研对象是谁、已知能做什么。没有这两份清单，Part 1 的场景矩阵和 Part 2 的用户验证都无从入手。
>
> **数据来源**：官方文档、GitHub README 和 Issues、官网、Discord/Reddit/X 社区整理、YouTube/Bilibili 教程。

---

### 0.1 OpenClaw 生态项目清单

> **格式**：以 `openclaw_ecosystem_projects.csv` 为主记录文件，便于数据库追踪。每新增项目时，先使用 `openclaw_project_record_template.md` 完整填写，再将关键字段同步至 CSV。
>
> **与 AI 协作说明**：将项目记录模板连同已收集资料（官网截图、README、社区帖子）一并提供给 AI，要求按字段逐一填写并标注来源。AI 填写后需人工核查"已知限制"字段，确保有原文佐证。

**CSV 字段说明**：

| 字段 | 说明 |
|------|------|
| project_name | 项目名称 |
| type | 核心项目 / 托管类 / 基础设施类 / 安全类 / 垂直应用类 |
| region | 主要运营地区（全球 / 美国 / 中国 / 欧洲 / 中东 / 日韩，可多值） |
| description | 功能边界一句话摘要 |
| supported_scenarios | 已知支持场景（逗号分隔） |
| known_limitations | 主要限制（逗号分隔，需有来源） |
| messaging_platforms | 支持的消息平台：WhatsApp / Telegram / Discord / Signal / 微信 / **飞书** / **Slack** / LINE / 其他 |
| supported_llm | 支持的 LLM（逗号分隔） |
| active_status | 活跃 / 维护中 / 停止维护 |
| pricing_model | 免费 / 订阅 / 按量 / 一次性 / Freemium |
| notes | 其他备注 |

**按地区分节说明**：

- **全球**：无特定地区归属的项目（已预填 7 个已知项目）
- **美国**：在美国有实质运营或主要用户基础的项目
- **中国**：本土托管服务、本地 LLM 适配、微信/飞书/钉钉集成项目
- **欧洲**：有 GDPR 合规声明或欧洲服务器部署的项目
- **中东**：阿拉伯语支持、本地数据主权合规的项目
- **日韩**：LINE / Kakao 集成、本地语言 LLM 适配的项目

同一产品若在多个地区有实质运营，CSV 中 region 字段填多个值，模板文档中"地区特殊情况"节单独说明差异。

### 0.2 OpenClaw 已知应用场景清单

> **格式**：以 `openclaw_known_scenarios.csv` 为主记录文件，便于数据库追踪。调研时直接在 CSV 中补充字段。
>
> **与 AI 协作说明**：将官方文档、社区帖子、教程截图提供给 AI，要求 AI 识别场景并按字段填写，每条标注来源链接。"已知卡点"字段需有原文引用，不接受 AI 推测。

**CSV 字段说明**：

| 字段 | 说明 |
|------|------|
| scenario_name | 场景名称（简洁命名） |
| task_category | 对应 Part 1 的任务类型：信息获取与处理 / 跨系统自动化 / 决策辅助 / 交互代理 / 个人知识管理 |
| description | 场景描述（用户做什么、Agent 做什么，1–2 句） |
| source | 数据来源类型：官方文档 / Discord / Reddit / X / 教程 |
| source_url | 来源链接 |
| region_us | 美国有实际使用记录：是 / 否 / 未知 |
| region_cn | 中国有实际使用记录：是 / 否 / 未知 |
| region_eu | 欧洲有实际使用记录：是 / 否 / 未知 |
| region_middle_east | 中东有实际使用记录：是 / 否 / 未知 |
| region_japan_korea | 日韩有实际使用记录：是 / 否 / 未知 |
| frequency_signal | 社区出现频率：高频 / 低频 / 单次 / 待调研 |
| required_skills | 所需核心 Skills（逗号分隔） |
| known_pain_points | 社区中提及的已知卡点（有原文佐证，逗号分隔） |

CSV 已预填 14 个初始场景，调研时补充完整字段并添加遗漏场景。

### 1.4 优先级矩阵推导

将 1.1 / 1.2 / 1.3 的数据叠加，生成优先级矩阵。

**推导逻辑**：

```
场景价值评分 = 用户需求强度 × 渗透潜力 × (1 / 竞争饱和度) × 可行性初判
```

> **说明**：此时用户验证尚未进行，"可行性初判"基于 Part 0 已知信息 + 社区反馈粗判，不依赖实测数据。"用户需求强度"基于 Part 0 社区频率信号初判，Part 2 用户验证后可修正。

**象限定义**：

| 象限 | 用户需求 | 竞品质量 | 可行性初判 | 结论 |
|------|----------|----------|------------|------|
| **优先做** | 强（真实高频） | 粗糙或无 | 可行 | 用户需求真实 + 竞争空白，重点投入 |
| **差异化做** | 强 | 已有但有缺口 | 可行 | 需要明确差异化角度才能进入 |
| **观察** | 弱或未验证 | 粗糙 | 部分可行 | 需等技术成熟或市场教育 |
| **不做** | 强 | 优秀 | — | 竞争饱和，无差异化空间 |
| **排除** | 任意 | 任意 | 结构性不可行 | 技术约束导致 |

**输出**：按优先级排序的场景列表，每个场景附优先级依据（1–2 句话）。这个列表同时作为 Part 2 轻量用户验证和 Part 3 针对性技术验证的输入——只对"优先做"和"差异化做"的场景进行实测。

## Part 2：轻量用户验证

> **目标**：带着 Part 1 的场景矩阵和优先级初判，快速验证场景真伪、发现遗漏、收集第一手卡点，并对 Part 1 的优先级排序进行修正。这是在做任何技术测试之前，先确认我们对用户需求的假设是否成立。
>
> **规模**：5–6 人，1 周内完成。

---

### 2.1 研究设计（快速验证版）

#### 受访者来源

- OpenClaw Discord 活跃用户
- Reddit r/selfhosted 发帖者
- Product Hunt 评论者

#### 筛选标准

- 曾使用过 OpenClaw 或同类 Agent 工具（至少 2 周以上）
- 有实际工作/生活场景在使用
- 优先覆盖不同地区背景（至少 2 个地区有代表）

#### 访谈时长

30 分钟（比完整版精简，聚焦场景验证）

---

### 2.2 访谈指南（精简版）

#### 模块一：背景（5 分钟）

- 你平时用 AI 工具主要做什么？大概从什么时候开始用？
- 你是怎么知道 OpenClaw 的？第一次用是什么时候？

#### 模块二：场景矩阵逐一验证（15 分钟）

> 拿着 Part 1 的场景矩阵，逐一与用户过。

- 这个场景你用过吗？
- 如果用过：多久用一次？最近一次是什么时候？
- 在用的过程中，卡在哪里了？（追问具体细节）
- 如果没用过：为什么没用？是不需要，还是试过但没成功？
- 你知道 XX 竞品吗？用过吗？和 OpenClaw 比怎么样？

#### 模块三：开放探索（10 分钟）

- 你实际最想让 OpenClaw 帮你做什么？
- 现在做不到的是什么？
- 如果它能帮你解决一个问题，你最希望是哪个？

---

### 2.3 输出

#### 场景真伪判定表

对 Part 0 中每个场景，根据访谈结果标注：

| 场景名称 | 判定 | 依据 |
|----------|------|------|
| — | 真实高频 / 真实低频 / 伪需求 / 新发现 | 几人确认、使用频率、具体描述 |

**判定标准**：
- **真实高频**：3 人以上确认在用，且频率为每周或更高
- **真实低频**：有人在用，但频率低于每月一次
- **伪需求**：无人确认实际使用，或试过但放弃
- **新发现**：Part 0 清单中没有，但访谈中被多人提到

#### 优先级矩阵修正建议

基于用户反馈，对 Part 1 的优先级排序进行修正：

| 场景名称 | Part 1 优先级 | 修正后优先级 | 修正依据 |
|----------|-------------|-------------|----------|
| — | 优先做 / 差异化做 / 观察 / 不做 | — | 用户反馈摘要 |

#### 真实卡点清单

用户原话记录，不做归因（归因在 Part 3 技术验证后进行）：

```
卡点描述：[用户原话]
所属场景：[场景名称]
提到人数：[X / 总访谈人数]
```

#### 遗漏场景补充

访谈中新发现的场景，补充回 Part 0 的场景清单 CSV。

## Part 3：针对性技术验证

> **目标**：只针对 Part 1 筛出的高优先级场景（经 Part 2 用户验证修正）（3–5 个），用 Part 2 用户访谈中收集的真实任务设计测试用例，验证技术可行性并定位卡点根因。不做全面技术基准测试，不做源码审计。

---

### 3.1 高优场景任务测试

基于 Part 2 用户访谈中收集的真实任务设计测试用例，每个高优场景设计 3–5 个代表性任务。

**测试用例设计原则**：
- 任务来自用户访谈中的真实描述，不用理论 L1–L5 分层模型
- 每个任务附用户原话作为来源说明
- 任务复杂度应覆盖该场景下的典型范围

**每个任务执行 20 次，记录**：
- 任务完成率（TCR）
- 首次成功率（无需用户干预）
- 失败类型分布（模型推理错误 / 工具调用失败 / 权限问题 / 超时）
- 平均 token 消耗
- 端到端延迟（P50 / P95）

**测试记录模板**：

```
场景名称：[来自 Part 1 高优场景]
任务描述：[用户访谈中的真实任务]
用户原话：[Part 2 收集的原始描述]
执行次数：20
TCR：X%
首次成功率：X%
失败类型分布：[分类统计]
平均 token 消耗：X
端到端延迟：P50 = Xs, P95 = Xs
```

---

### 3.2 安全边界评估

> 仅评估高优场景涉及的安全维度，不做全面安全审计。

根据 3.1 测试的场景，评估以下适用维度：

| 评估维度 | 适用条件 | 调研内容 | 产品决策依据 |
|----------|----------|----------|-------------|
| Prompt Injection 风险 | 场景涉及处理外部数据（邮件/网页） | 实测：恶意内容能否劫持 Agent 行为 | 哪些场景不能做无人值守自动化 |
| 敏感数据处理 | 场景涉及个人/企业敏感信息 | Agent 执行链中敏感数据是否意外进入 LLM 上下文 | 隐私场景的合规约束 |
| 本地 vs 云端数据流 | 场景有数据隐私需求 | 哪些数据留本地，哪些发往 LLM API | 隐私敏感用户的产品设计约束 |
| Skills 权限范围 | 场景涉及系统级操作 | Skills 的系统访问范围是否可控 | 企业场景的技术可行性前提 |

---

### 3.3 卡点根因分类

对 3.1 中每个失败案例进行根因归因：

| 根因类型 | 描述 | 产品含义 |
|----------|------|----------|
| 模型推理不稳定 | 相同输入不同输出，幻觉，步骤遗漏 | 需要输出验证层或人工确认节点 |
| Skill 设计缺陷 | SKILL.md 表达能力不足，参数传递错误 | 需要更结构化的 Skill 规范 |
| 权限与沙箱限制 | 所需权限被拒，沙箱隔离过强 | 需要细粒度权限配置 UI |
| 消息 UI 表达瓶颈 | 用户无法在消息界面清晰表达复杂意图 | 需要引导式输入或任务模板 |
| 错误处理缺失 | 无重试机制，失败后无明确反馈 | 需要错误处理和恢复流程设计 |
| 本地模型能力差距 | 本地模型在复杂任务上与云端差距大 | 需要明确本地 vs 云端的任务分配策略 |

**输出**：每个高优场景的卡点根因分布，作为 Part 4 深度用户研究和 Part 5 产品定义的输入。

### 4.2 访谈指南（含场景专项模块）

> 以下是通用访谈结构，每个场景需在「场景专项探索」部分补充具体问题。与 Part 2 的精简版不同，此轮访谈可以带着 Part 3 的技术验证数据，探讨用户对技术限制的真实容忍度。

#### 模块一：背景与上下文（5 分钟）

- 你平时用 AI 工具主要做什么？大概从什么时候开始用？
- 你是怎么知道 OpenClaw 的？第一次用是什么时候？

#### 模块二：使用行为还原（15 分钟）

> 目标：还原真实的使用过程，而非听用户总结评价。

- 上次用 OpenClaw 完成了什么任务？你能描述一下整个过程吗？（逐步追问）
- 在这个过程中，哪一步你等待的时间最长？
- 哪一步你最不确定 AI 会不会做对？
- 有没有哪个地方你本来想让它做，但最后没让它做——为什么？
- 这个任务最终有没有完成？如果没有，是在哪一步停下来的？

#### 模块三：场景专项探索（15 分钟）

> 根据 Part 1 的具体场景补充。此模块的关键差异：可以带着 Part 3 的技术数据向用户验证容忍阈值。

**通用技术约束验证问题**（基于 Part 3 数据填入具体数字）：
- 如果这个任务的成功率是 [Part 3 实测 TCR]%，你还会用吗？
- 如果需要你确认某个步骤才能继续，你能接受吗？多频繁的确认你能忍受？
- 如果执行时间需要 [Part 3 实测延迟] 秒，你觉得可以接受吗？

**如果是「信息获取与监控」场景**：
- 你目前是怎么追踪（某类信息）的？AI 介入前你是怎么做的？
- 你希望 AI 收集完信息后直接告诉你结论，还是给你原始数据？为什么？
- 如果它拿回来的信息有 10% 是错的，你能接受吗？如何发现错误？

**如果是「跨系统自动化」场景**：
- 你现在跨多少个工具完成这个任务？哪个工具之间的衔接最让你头疼？
- AI 执行这个任务时，你觉得哪些步骤你必须自己确认，不放心让它自动做？
- 任务执行到一半失败了，你怎么知道失败了？失败后你怎么处理？

#### 模块四：卡点深挖（10 分钟）

> 针对模块二中出现的卡点逐一追问根因。

- 你说这里会出错——上次出错的时候，具体发生了什么？
- 你觉得这个问题是 AI 不够聪明，还是这个工具设计有问题，还是你没说清楚？
- 如果这个问题解决了，你会更频繁地用这个功能吗？频繁多少？

#### 模块五：付费意愿与价值感知（5 分钟）

- 你现在用 OpenClaw 是免费还是付费？
- 如果有一个产品能把刚才说的问题都解决，你愿意付多少钱？（追问：为什么是这个数字？）
- 你现在不用 AI 的话，这件事是怎么解决的？花多少时间/钱？

### 5.3 交互流程设计框架

> Agent 产品的交互设计需要同时描述**用户侧**（用户看到什么、做什么）和**Agent 侧**（Agent 在做什么、状态如何）。

#### 双轨交互流程模板

```
用户侧                          Agent 侧
──────────────────────────────────────────────────────
[用户输入意图]                →  [意图解析 + 任务规划]
                              ←  [展示执行计划，等待确认]
[用户确认 / 修改]             →  [开始执行]
                              ←  [进度更新：正在做第 X 步]
                              ←  [遇到需要确认的节点：展示选项]
[用户做出选择]                →  [继续执行]
                              ←  [任务完成：展示结果摘要]
[用户审核结果]                →  [用户接受 / 要求修改]
```

#### 关键设计决策点

每个交互流程需要对以下问题做出明确决策，不能模糊处理：

| 决策点 | 需要回答的问题 |
|--------|---------------|
| **透明度** | 用户需要看到 Agent 的执行步骤吗？看多少细节？ |
| **确认粒度** | 哪些步骤需要确认？确认太多影响体验，太少影响信任 |
| **进度呈现** | 长时任务（>30 秒）如何让用户感知进度？ |
| **失败呈现** | 任务失败时展示什么？用户能做什么？ |
| **撤销机制** | 哪些操作可以撤销？撤销的时间窗口是多少？ |
| **通知策略** | 异步任务完成时如何通知用户（哪个渠道、什么时机）？ |

#### 地区差异说明

交互流程需要在以下维度标注地区差异：
- **确认节点**：日本用户对出错容忍度低，需要更多确认节点；美国用户更接受自动化
- **语言与 LLM**：中文/阿拉伯语/日语任务需要评估对应语言 LLM 的能力差距，影响 TCR
- **消息平台**：中国的微信限制、日本的 LINE 接口特性，影响消息 UI 的表达能力

**常见约束类型及产品含义**（调研后填入实测数据）：

|----------|----------|
| 跨系统任务 TCR 不足 | 此类任务不能做无人值守，必须加确认节点 |

| 无沙箱隔离 | 企业多用户场景下 Skills 权限管理需要在产品层实现 |

### 5.5 设计原则提炼方法

> 设计原则不是凭空写出来的，而是从卡点根因中推导出来的。以下是推导流程。

#### 推导流程

```
卡点（来自 Part 4 步骤一）
    ↓ 追问：这个卡点反映了什么深层设计问题？
根因（对照 Part 3 分类）
    ↓ 追问：如果要从设计层面缓解这个根因，核心原则是什么？
设计原则（1 句话，行为导向）
    ↓ 落地到：哪些功能/流程应用这条原则？
具体设计决策
```

#### 示例推导

```
卡点：「用户不知道任务执行失败了，等了很久才发现」（出现 6/10 次访谈）
根因：错误处理缺失 + 进度状态不透明
设计原则：「Agent 的任何状态变化，用户都应在 30 秒内得到反馈」
具体决策：
  - 执行超过 10 秒的任务必须展示进度条或步骤进展
  - 任务失败必须通过消息主动告知用户，不依赖用户主动查询
  - 失败消息必须包含：失败原因（非技术语言）+ 用户可做的下一步
```

#### 设计原则记录格式

```markdown
## 原则名称：[简洁命名，如「状态透明」]

### 来源
- 卡点：[来自哪个访谈，什么场景]
- 出现频率：[X/Y 次访谈]
- 根因类型：[对照 Part 3 分类]

### 原则陈述
[一句话，行为导向，可测试]

### 适用范围
[哪些功能/流程需要遵守这条原则]

### 反例（什么是违反这条原则的）
[具体描述，帮助团队理解边界]

### 优先级
高 / 中（高 = 违反会直接损害信任或导致放弃使用）
```
